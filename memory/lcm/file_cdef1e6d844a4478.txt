## 00:02 RemoteAssistService 录制流程修复

### Bug: TouchRecordOverlay 未激活导致录制为空
- **根因**: 用户点"好的，开始"后直接调 `requestMediaProjection()` 弹系统弹窗，但没有先启动 MediaProjection 录制。TouchRecordOverlay 没激活，弹窗上用户的触摸操作完全没被拦截录制。
- **修复**: 在 `requestMediaProjection()` 之前先调 `RemoteAssistService.instance?.startMpRecording()`，激活 TouchRecordOverlay 拦截触摸坐标。

### RemoteAssistService API 注意事项
- `getInstance()` 是错误的调用方式，正确的是 `instance` 属性（Kotlin object 单例的 @JvmStatic companion 属性）
- 用 `RemoteAssistService.instance?.xxx()` 而非 `RemoteAssistService.getInstance()?.xxx()`

### 修复后的完整流程
1. 点"允许" → 录制引导弹窗
2. 点"好的，开始" → **先启动录制(Overlay激活)** → 再弹出 MediaProjection 系统弹窗
3. 用户操作弹窗 → Overlay 拦截每次触摸坐标 → 录进 recordedSteps
4. 授权成功 → `onActivityResult` → 保存录制

## 00:14 录制永不结束 + Toast 不显示修复

### Bug1: 录制永不结束，手机卡住
- **根因**: 首次录制流程里，弹系统弹窗后没有显示 `showLoading()`（含"完成录制"按钮）。用户操作完弹窗直接进 `startScreenCapture()`，TouchRecordOverlay 一直没关，持续拦截所有触摸导致手机卡。
- **修复**: 弹系统弹窗后立即调 `showLoading()`，让用户看到"完成录制"按钮
- **兜底**: `onActivityResult` 成功/失败都先调 `stopMpRecording()` 关 Overlay

### Bug2: Toast 不显示
- **根因**: AccessibilityService 的回调不在主线程，`Toast.makeText().show()` 静默失败。
- **修复**: 所有 Toast 包在 `handler.post { }` 里

### 修复后测试流程
1. 子女端发请求 → 老人端点"允许"
2. 录制引导 → 点"好的，开始"
3. 应该看到 Toast「🎬 录制中...」+ Loading 界面（带"完成录制"按钮）
4. 手动操作弹窗授权
5. 授权成功自动进协助界面（录制也会自动停止）

## 00:20 saveRecording() 未重置 KEY_AUTO_DISABLED 导致二次回放仍被禁用

### Bug: 重新录制成功后，第二次触发仍走"自动授权已禁用"分支
- **根因**: `PermissionRecordManager.saveRecording()` 保存新录制时没有重置 `KEY_AUTO_DISABLED` 和 `KEY_REPLAY_FAIL_COUNT`。之前测试自动回放失败3次触发了自动禁用，重新录制后禁用标志仍为 true。
- **修复**: 在 `saveRecording()` 成功后追加：
  ```kotlin
  prefs.edit()
      .remove(KEY_AUTO_DISABLED)
      .remove(KEY_REPLAY_FAIL_COUNT)
      .apply()
  ```
- **验证**: 用户需重新录制 → 第二次发请求验证自动回放流程

### 涉及文件
- `PermissionRecordManager.kt` — `saveRecording()` 方法，v11 修改
- 已有 `clearRecording()` 和 `resetAutoDisabled()` 方法，但 `saveRecording()` 成功时没自动调用

## 00:30 录制坐标误录修复 + 取消自动禁用

### Bug: 录到的坐标全是 App UI 点击，不是弹窗按钮
- **根因**: TouchRecordOverlay 在 `requestMediaProjection()` 之前就开始录制，用户点"好的，开始"按钮的触摸被录进去（step 0: (1085, 2401) 是"好的，开始"按钮坐标）
- **K70 屏幕分辨率**: 1440×3200，录到的坐标全在下半屏（App UI 区域）
- **连锁问题**: 回放失败3次 → 自动禁用 → 重置又被覆盖

### Fix 1: 录制忽略前1秒触摸
- `recordTouchCoord()` 加 1000ms 冷启动期，`System.currentTimeMillis() - recordingStartTime < 1000` 时直接 return
- 跳过"好的，开始"按钮的误录

### Fix 2: 取消自动禁用
- 删除 `PermissionRecordManager` 中的自动禁用逻辑（回放失败不再锁死，允许无限重试）
- `tryAutoHandle()` 失败只 log 不禁用
- `startPermissionFlow()` 不再检查 `isAutoDisabled`
- `saveRecording()` 不再清除 KEY_AUTO_DISABLED（因为不再有禁用逻辑）

- **根因**: `PermissionRecordManager` 运行在 AccessibilityService（独立进程），无法直接调用 Activity 方法。回放成功后 Activity 不知道，30秒倒计时继续跑
- **解决方案**: 广播机制
  ```
  PermissionRecordManager.onReplaySuccess()
    → sendBroadcast(ACTION_AUTO_PERMISSION_GRANTED)
        ↓
  RemoteAssistActivity.registerAutoPermissionReceiver()
    → 收到广播 → cancel countdown → startScreenCapture()
  ```

### 同修: getDialogSignature() 对 HyperOS 弹窗失效
- **根因**: `getDialogSignature()` 依赖 `window.title` 关键词，但 HyperOS MediaProjection 弹窗属于 `com.android.systemui`，`window.title` 永远为 null
- **修复**: 过滤前先用 `hasPermissionDialog()` 扫描节点确认是权限弹窗，再用节点内容（alertTitle + button1Text）生成签名

## 09:13 v14 回放成功后倒计时卡住 — 广播机制修复

### Bug: 回放成功后 RemoteAssistActivity 倒计时继续，不进入协助界面
- **根因**: `PermissionRecordManager` 运行在 AccessibilityService（独立进程），无法直接调用 Activity 方法。回放成功后 Activity 不知道，30秒倒计时继续跑
- **解决方案**: 广播机制
  ```
  PermissionRecordManager.onReplaySuccess()
    → sendBroadcast(ACTION_AUTO_PERMISSION_GRANTED)
        ↓
  RemoteAssistActivity.registerAutoPermissionReceiver()
    → 收到广播 → cancel countdown → startScreenCapture()
  ```

### 同修: getDialogSignature() 对 HyperOS 弹窗失效
- **根因**: `getDialogSignature()` 依赖 `window.title` 关键词，但 HyperOS MediaProjection 弹窗属于 `com.android.systemui`，`window.title` 永远为 null

**问题1：倒计时卡（还没点允许就卡）**

- **修复**：2s 后开始轮询，每 1s 检查一次，最多等 6s，等弹窗真正消失才成功

**问题3：回放后手机卡**
- **根因**：回放结束后 `cleanupReplayState()` 清了 handler，但 `onAccessibilityEvent` 继续触发，`hasPermissionDialog()` 仍在每帧扫描
- **修复**：回放期间（`isReplaying=true`）直接跳过 `hasPermissionDialog()` 检测，完全不扫描

### 🔑 v17 关键发现：hasPermissionDialogByText() 误报（根因定位）

**核心误报 Bug**：`hasPermissionDialogByText()` 扫描所有窗口的节点找 BUTTON_TEXT_KEYWORDS（"允许","取消","开始"等），但 **RemoteAssistActivity 自身的 UI 就包含"允许"和"取消"按钮**，扫描到应用自身窗口时误判为权限弹窗。

**三个问题的统一因果链**：
1. **倒计时卡**：误报 → `tryAutoHandle()` → `dispatchGestureClick()` → `CountDownLatch.await(2, TimeUnit.SECONDS)` 阻塞主线程 2 秒
2. **步骤2/3失败**：步骤1坐标是录制时 MediaProjection 弹窗的坐标，但当前屏幕是 RemoteAssistActivity，点击无效；即使第1步偶然成功，`waitForDialogChangeAndReplay()` 检测到弹窗消失就调 `onReplaySuccess()` 跳过剩余步骤
3. **回放后手机卡**：`onReplayFailed()` 后 `isReplaying=false`，`onAccessibilityEvent` 继续触发，误报循环 → 每次循环阻塞主线程 2 秒

**补充发现**：
- `hasPermissionDialogByText()` 中用 `window.root?.packageName` 取包名，但 root 是节点树根节点，其 packageName 可能是被捕获的应用而非窗口本身的包名。应改用 `window.packageName`（AccessibilityWindowInfo 的字段）
- 进程隔离推论修正：AccessibilityService 与 Activity 虽不同进程，但 Service 的 `onAccessibilityEvent` 主线程阻塞确实影响 UI（同一进程组件共享主线程），dispatchGestureClick 内的 latch.await() 阻塞 Service 主线程是倒计时卡顿的直接原因

### v17 修复方案（待实施）

**修复1：过滤应用自身窗口（核心）**

**修复2：dispatchGestureClick 异步化**
