## 远程协助三Bug修复 v24（17:13-17:27）

### 三个Bug及修复

**Bug ①：点击"允许"无响应**
- 根因：`handleNewRequest()` 返回 false（重复请求）时跳过 `initViews()`，按钮监听器未设置
- 修复：v23 — 始终调 `initViews()`，无论是否新请求
- 文件：`RemoteAssistActivity.kt` 的 `onCreate()` 和 `onNewIntent()`

**Bug ②：自动允许开关失效**
- 根因：重复请求路径直接 `showRequestDialog()`，跳过 `auto_allow_assist` 检查
- 修复：v24 — 重复请求也检查 `auto_allow_assist`，autoAllow=true 时走 `onAllowClicked()`
- 文件：`RemoteAssistActivity.kt` 的 `onCreate()` 和 `onNewIntent()` 的 `!isNewRequest` 分支

**Bug ③：无障碍设置跳转不到详情页**
- 根因：`openAccessibilitySettings()` 只返回通用列表页 Intent
- 修复：v24 — 先尝试 `ACTION_ACCESSIBILITY_DETAILS_SETTINGS` 跳详情页，`resolveActivity` 失败则 fallback 通用列表；`startActivity` 异常也 fallback
- 文件：`RemoteAssistManager.kt` 新增 `openAccessibilitySettings()` 方法 + `RemoteAssistActivity.kt` 中调用处的 try-catch fallback 链
- **关键**：`ACTION_ACCESSIBILITY_DETAILS_SETTINGS` 和 `EXTRA_ACCESSIBILITY_COMPONENT_NAME` 是 Android API 24+ 隐藏常量，Kotlin 不能直接引用，需用字符串 `"android.settings.ACCESSIBILITY_DETAILS_SETTINGS"` 和 `"android.intent.extra.ACCESSIBILITY_COMPONENT_NAME"`
- 需要额外 import：`android.content.ComponentName`

## 远程协助三大Bug修复 v25（17:42-17:58）

### Bug ①：子女端只看到一帧就卡住
**根因**：`network_security_config.xml` 中写了 `<domain includeSubdomains="true">192.168.0.0</domain>`，以为能覆盖整个网段。但 Android 的 `domain-config` 做域名精确匹配，不支持 CIDR 网段 → `http://192.168.4.19` 被当成 Cleartext HTTP 拒绝 → `ScreenCaptureService.notifyScreenReady()` 抛 `IOException: Cleartext HTTP traffic to 192.168.4.19 not permitted` → 进程崩溃 → 帧上传全部停止。

**教训**：Android `network_security_config.xml` 的 `domain-config` 不支持 CIDR，必须写具体IP或域名。

### Bug ②：再次请求老人端闪退
**根因**：同上！`notifyScreenReady()` 在 `thread {}` 中执行但未 try-catch → `IOException` 直接崩溃整个进程 → App重启后状态丢失 → 再次请求时状态混乱。

**修复**：`notifyScreenReady()` 内部加 try-catch，异常时 Log.e 而非崩溃。

**教训**：Kotlin `thread {}` 中未捕获异常会崩溃整个进程，必须 try-catch。

**修复**：
- v25: 重复请求（`!isNewRequest`）只显示当前状态，不走 `onAllowClicked()`
- `onAllowClicked()` 中 `respondToRequest` 失败时：如果已在协助中则忽略（不toast），否则显示友好提示（不是后端原始错误消息）

## v26 远程协助核心Bug修复 — 无限循环弹窗 + 屏幕共享中断（18:05-18:14）

### Bug ①：每30秒弹一次协助请求（无限循环弹窗）

**根因**：`RemoteAssistManager.kt` 的 WS 监听器用 `fromId + currentTimeMillis()/1000` 做请求去重ID。`System.currentTimeMillis()/1000` 每秒都变 → `lastNotifiedRequestId` 永远匹配不上 → 每次WS推送的请求都当新请求 → 弹 Activity → 15秒自动拒绝 → 关闭 → 健康检查30秒后重启轮询 → 又收到同一请求 → 又弹 → 无限循环。

**时间线证据**：17:54~18:01，每30秒弹一次 RemoteAssistActivity（共15次）。

**修复**：
- WS 监听器：`guardianId + requestTime` 替代 `fromId + currentTimeMillis()/1000`
- HTTP poll：`fromId + requestTime` 替代 `fromId + remainingSeconds`（remainingSeconds每秒递减也导致去重失效）

**教训**：去重ID必须用请求本身的唯一标识（如requestTime），绝不能用当前时间戳。

### Bug ②：ScreenCaptureService 18秒后被杀

**根因**：`RemoteAssistActivity.onDestroy()` 中 `if (isAssisting) cleanupAssist()` → `stopService(ScreenCaptureService)` → 屏幕共享中断。Activity 因自动拒绝倒计时/系统回收而关闭 ≠ 用户想结束协助。

**修复**：`onDestroy()` 中 isAssisting=true 时只解绑 Service 不停 Service，让 ScreenCaptureService 继续运行。Service 应由用户主动"结束"或对方断开来停止。

**教训**：ScreenCaptureService 的生命周期应独立于 Activity。Activity 是 UI 层，Service 是业务层。

### Bug ③：autoAllow 在重复请求时失效（v25 回归）

**根因**：v25 修复 Bug ③（重复请求弹"没有待处理请求"）时，把重复请求路径改成了只显示状态，删除了 autoAllow 检查。

**修复**：v26 恢复 autoAllow 检查：重复请求时如果 autoAllow=true，走 `onAllowClicked()`。
