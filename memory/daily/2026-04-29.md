## signedVelocity 方向分析结果（12:31-12:44）

### 代码分析
FallDetector.kt 中 signedVelocity 计算逻辑：
```kotlin
val accel = when {
    accMag < 0.7f -> 9.81f * (1.0f - accMag)        // FF: 正加速（向下）
    accMag > 1.3f -> -9.81f * (accMag - 1.0f) * 0.5f // 超重: 负加速（向上）
    else -> 9.81f * (1.0f - accMag)                   // 正常: 微正/微负
}
signedVelocity += accel * dt_s
```

### 结论：有方向信息 ✅
- 跌倒(FF) → signedVelocity 增加（正方向=向下加速）
- 推手机(超重) → signedVelocity 减少（负方向=向上加速）
- 但幅度不对称：FF段 accel 可达 +6.87，推手机过渡段(1.0-1.3g) accel 只有 -0.98 ~ -1.96

### 漏洞修复评估
| 漏洞 | 结论 |
|------|------|
| 1. 推手机误合并 | ✅ 可修：delta<0 = 推手机，delta>0 = 跌倒 |
| 2. 速度方向 | ✅ 有方向信息 |
| 3. 积分漂移 | ⚠️ 仍在（冷却期重置但间隔期内可能漂移）|
| 4. 数据不足 | ⚠️ 仍在（短间隔<10帧无法算均值）|
| 5. 阈值敏感 | ✅ 改善（有方向后更可靠）|

3. MIUI 系统强杀进程（isRunning 标志未重置，但传感器从未初始化）
4. 用户再看APP → "守护已开启"（误判）→ 实际检测功能没在跑 ❌

**根本原因**：MIUI 强杀后，BOOT_COMPLETED 广播只能恢复一次，无法持续保活。FallDetectionService 被杀后没有自动拉起来的机制。

### 解决方案
引入 **WorkManager 定时心跳保活**：每 15 分钟自动把服务拉起来。

### 改动内容

#### 1. build.gradle.kts
添加 `androidx.work:work-runtime-ktx:2.9.0`

#### 2. BootWorker.kt（新建）
- 继承 `Worker`，`doWork()` 里调用 `startForegroundService(FallDetectionService)`
- 定时策略：`PeriodicWorkRequestBuilder<BootWorker>(15, TimeUnit.MINUTES)`，flexInterval=5分钟
- 支持 `CANCEL_AND_REENQUEUE`：每次重置周期，防止被系统优化掉
- 如果 `auto_start=false`，跳过启动
- **只恢复服务，不触发警报**（isAlertFired 状态保持）

#### 3. HomeFragment.kt
- 点击"开启守护"→ 同时调用 `BootWorker.schedule(context)` 安排心跳
- 点击"关闭守护"→ 同时调用 `BootWorker.cancel(context)` 取消心跳

#### 4. BootReceiver.kt（已有逻辑不变）
保留用于**首次开机**恢复，后续保活靠 WorkManager

### 工作流程
```
用户打开APP → startService() + WorkManager.schedule()（15分钟心跳开始）
       ↓
MIUI强杀 → FallDetectionService 被杀，isRunning=false
       ↓
15分钟后 → BootWorker.doWork() → startForegroundService() → 服务恢复
       ↓
用户再打开APP → 看到"守护已开启" ✅
```

## 15:27 - 开机自启根本原因确认（pre-compaction记录）

### 最终诊断

## 15:40 - v0.30.5 守护服务保活方案（WorkManager 心跳）

### 问题根源
用户描述：APP打开时显示"守护已开启"，但系统显示服务未运行。

**完整场景**：
1. 用户打开跌倒宝 → FallDetectionService 启动 ✅
2. 用户把APP切到后台/锁屏
3. MIUI 系统强杀进程（isRunning 标志未重置，但传感器从未初始化）
4. 用户再看APP → "守护已开启"（误判）→ 实际检测功能没在跑 ❌

**根本原因**：MIUI 强杀后，BOOT_COMPLETED 广播只能恢复一次，无法持续保活。FallDetectionService 被杀后没有自动拉起来的机制。

### 解决方案
引入 **WorkManager 定时心跳保活**：每 15 分钟自动把服务拉起来。

### 改动内容

#### 1. build.gradle.kts
添加 `androidx.work:work-runtime-ktx:2.9.0`

#### 2. BootWorker.kt（新建）
- 继承 `Worker`，`doWork()` 里调用 `startForegroundService(FallDetectionService)`
- 定时策略：`PeriodicWorkRequestBuilder<BootWorker>(15, TimeUnit.MINUTES)`，flexInterval=5分钟
- 支持 `CANCEL_AND_REENQUEUE`：每次重置周期，防止被系统优化掉
- 如果 `auto_start=false`，跳过启动
- **只恢复服务，不触发警报**（isAlertFired 状态保持）

#### 3. HomeFragment.kt
- 点击"开启守护"→ 同时调用 `BootWorker.schedule(context)` 安排心跳
- 点击"关闭守护"→ 同时调用 `BootWorker.cancel(context)` 取消心跳

#### 4. BootReceiver.kt（已有逻辑不变）
保留用于**首次开机**恢复，后续保活靠 WorkManager

### 工作流程
```
用户打开APP → startService() + WorkManager.schedule()（15分钟心跳开始）
       ↓
MIUI强杀 → FallDetectionService 被杀，isRunning=false
       ↓
15分钟后 → BootWorker.doWork() → startForegroundService() → 服务恢复
       ↓
用户再打开APP → 看到"守护已开启" ✅
```
