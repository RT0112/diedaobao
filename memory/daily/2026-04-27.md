## 2026-04-27 22:00 速度大趋势FF合并 v0.27 Python验证通过

### 用户新思路
用**速度大趋势**替代超重检测判断FF合并：
- 真跌倒：速度大趋势单调递增（小→大→碰撞骤降）
- 拿起放下：速度反复震荡（高→低→高→低）
- 缓慢跌倒：趋势较弱但仍递增

### 实现方案

**算法**：多层速度趋势检测（决策链层）

```
指标1: Spearman秩相关（单调性）
指标2: 一阶差分正比例（递增比例）
指标3: 短期MA vs 长期MA（近期趋势）
指标4: 峰值位置（真跌倒峰值在末尾）

综合评分 = rho*0.25 + pos_ratio*0.35 + MA_ratio*0.5 + peak_pos*0.15

判断: score>0.4 或 pos_ratio>0.55 或 (rho>0.3 且 pos_ratio>0.5)
```

**架构变更**：
- 删除所有超重检测逻辑（超重阈值、超重时长、硬切分等）
- FF合并只基于时间邻近性（500ms）
- 速度趋势检测在决策链中运行
- 如果速度趋势不是递增的（震荡模式），排除

## 2026-04-27 22:07 速度趋势FF合并 - 场景7修复

### 问题发现
场景7（真跌倒+暂停）的FF合并判断时机有误：
- 第一段FF结束 → 清空velocityHistory
- 第二段FF开始 → 检查velocityHistory（已被清空）→ 默认合并
- 但速度趋势检测在FF**结束**时进行，此时历史还在

### 修复方案
速度趋势检测应在**FF段结束时**进行：
1. 保留velocityHistory直到新FF段确认合并或硬切分
2. FF段结束时判断趋势，如果是震荡模式则设置不合并标志
3. 或者：不清空velocityHistory，让下一段FF开始时能检查历史

### 当前状态
- 8场景全部通过 ✅（包括场景7）

- 需要调整：合并判断应在第一段FF结束时进行

### detectVelocityTrend() 算法实现

```kotlin
// Spearman秩相关
val rho = calculateSpearmanCorrelation(indices, velocityHistory)

// 一阶差分正比例
val positiveRatio = positiveCount.toFloat() / (velocityHistory.size - 1)

// 短期MA vs 长期MA
val shortTermMA = velocityHistory.takeLast(4).average()
val longTermMA = velocityHistory.average()
val maRatio = if (longTermMA != 0.0) shortTermMA / longTermMA else 1.0

// 峰值位置
val peakIndex = velocityHistory.indices.maxByOrNull { velocityHistory[it] } ?: 0
val peakPosition = peakIndex.toFloat() / (velocityHistory.size - 1)

// 综合评分
val trendScore = rho * 0.25f + positiveRatio * 0.35f + maRatio.toFloat() * 0.15f + peakPosition * 0.15f

// 判断逻辑
val shouldMerge = trendScore > 0.4f || positiveRatio > 0.55f || (rho > 0.3f && positiveRatio > 0.5f)
```

### 帧数不足处理
当 `velocityHistory.size < 4`：
- FF < 100ms → 不合并（保守）
- FF >= 100ms → 合并

## 2026-04-27 22:55 完整算法逻辑文档整理

### 算法架构总览

**状态机**：
```
MONITORING ──(冲击触发)──> IMPACT_DETECTED ──(1500ms确认期)──> 决策链 ──> MONITORING
     │                          │
     │                          └─> 方差>0.5 取消 ──> MONITORING
     │
     └─> FF追踪（持续运行）
```

### 核心分支逻辑

1. **冷却期检查**：now - lastPeakTime < 2800ms → 跳过所有处理
2. **FF追踪**（MONITORING状态）：
   - accMag < 0.7g → 进入/继续FF段，记录velocityHistory
   - accMag >= 0.7g → FF段结束，开始暂停期
   - 新FF段开始时：用速度趋势判断是否合并上一段FF
3. **冲击检测**：
   - dynamicAcc > 0.6g → 检查FF时长
   - FF in 1..199ms 且 冲击<5g → 立即排除，不进确认期
   - 否则进入IMPACT_DETECTED状态
4. **确认期**（1500ms）：
   - 方差 > 0.5 → 取消（用户在移动）
   - 超时 → 运行决策链
5. **决策链**：
   - ML推理（ONNX优先）
   - 物理评分 = FF×35% + 冲击×35% + 速度×30%
   - 加权评分 = 物理×36% + ML×64%
   - 三路径判断：ML高阈值 / ML低阈值+加权 / 物理覆盖

### 速度趋势检测算法（v0.27核心）

**四层指标**：
1. Spearman秩相关 rho（单调性）：真跌倒≈1，拿起放下≈0
2. 正差分比例：真跌倒≈1.0，拿起放下≈0.5
3. 短期MA/长期MA：真跌倒>1，拿起放下≈1
4. 峰值位置：真跌倒≈1.0（末尾），拿起放下≈0.5（中间）

**判断逻辑**：
```
trendScore = rho×25% + posRatio×35% + MA项 + peakPos×15%
isTrending = trendScore>0.4 || posRatio>0.55 || (rho>0.3 && posRatio>0.5)
```

**帧数不足处理**：
- FF < 100ms → 不合并（保守）
- FF >= 100ms → 合并

|------|-----|

| FF合并间隔 | 500ms |
| 确认期 | 1500ms |

| 方差取消 | 0.5 |
