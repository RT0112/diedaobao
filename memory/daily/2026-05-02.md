## 子女端双按钮闪退 — 根因已定位

### 崩溃根因（代码级确认）

**两个独立问题，均在 MapActivity：**

#### 问题1：WebView.destroy() 后继续访问（主因）
`scheduleJsSync()` 启动无限循环协程，Activity 销毁（`onDestroy` 调用 `webView.destroy()`）后协程仍在运行，导致崩溃。

```kotlin
// 修复：存储 Job 引用，onDestroy 取消
private var jsSyncJob: Job? = null

// onCreate 中：
jsSyncJob = lifecycleScope.launch {
    while (true) {  // ← Activity 销毁后仍继续 → crash
        webView.evaluateJavascript(...)
    }
}

// onDestroy 中添加：
jsSyncJob?.cancel()
```

#### 问题2：lifecycleScope.launch 回调在 Activity 销毁后执行
`webView.setWebViewClient` 中的 `onPageFinished` 和按钮点击回调中直接使用 `lifecycleScope.launch`，Activity 销毁后回调触发时 WebView 已不安全。

#### 问题3：CrashHandler 异常未持久化
日志只写入 Logcat（内存），App 重启后丢失，无法分析真实崩溃原因。

2. **import**：添加 `import kotlinx.coroutines.Job`

## MapActivity 深度代码审查 — 发现额外闪退根因

### 新发现的问题（v0.7.3 仍可能闪退）

#### 问题4：setContentView 被调用两次（严重）
MapActivity.onCreate 中：
1. 第79行先调用 `setContentView(webView)` — 把 webView 设为根内容
2. 第134行又调用 `setContentView(rootLayout)` — 替换为 rootLayout

rootLayout 包含同一个 webView 实例。第二次 `setContentView` 时，webView 已经是 DecorView 的子 View，把它移到 rootLayout 中可能导致布局异常和闪退。

**修复**：删除第79行的 `setContentView(webView)`，只保留最后 `setContentView(rootLayout)`。

#### 问题5：AndroidBridge vs Android 命名不一致
- `addJavascriptInterface(FenceSaveCallback(), "Android")` 注册名为 `"Android"`
- `scheduleJsSync()` 中调用 `window.AndroidBridge.syncFenceValues()` — 但 JS 中 `AndroidBridge` 是纯 JS 对象，**没有 `syncFenceValues` 方法**
- `syncFenceValues` 是 Java 注入的方法，挂在 `window.Android` 下
- 正确调用应为 `window.Android.syncFenceValues()`，而非 `window.AndroidBridge.syncFenceValues()`

**影响**：`addFenceLat/addFenceLng` 始终为 0.0，原生保存按钮会提示"请先在地图上选择围栏位置"。

#### 问题6：addJavascriptInterface 在 MIUI 上的安全限制
MIUI/某些国产 ROM 的 WebView 可能拦截 `addJavascriptInterface`，导致 `window.Android` 不存在。
JS 中有 `if (window.Android)` 检查所以不会 JS 报错，但保存功能完全失效。

// 旧方案：只读 GPS 缓存

// 新方案：强制新鲜 fix

## 22:40 — 围栏缓存永不过期 Bug 修复 ✅

### 用户反馈
"围栏改动后，老人端什么时候生效？"

### 根因
`FallDetectionService.kt` 中 `refreshFenceCache()` 只在 `startPullPolling()` 启动时调了**一次**：
```kotlin
private fun startPullPolling() {
    refreshFenceCache()  // ← 唯一一次调用！
    while (isActive) {
        pollPullRequest()  // 只 poll 位置，不碰围栏
        delay(30_000)
    }
}
```
`FENCE_REFRESH_INTERVAL = 30 * 60_000L` 常量定义了但**从未被使用**。
子女端增/删/改围栏 → 老人端完全不知道，必须杀进程重开。

### 修复
在 poll 循环中加定期检查，每 30 分钟自动刷新围栏缓存：
```kotlin
while (isActive) {
    // 每30分钟刷新围栏缓存 ← 新增
    if (System.currentTimeMillis() - lastFenceRefreshTime >= FENCE_REFRESH_INTERVAL) {
        refreshFenceCache()
    }
    pollPullRequest()
    delay(30_000)
}
```

## 23:00 子女端围栏权限不足问题（根因分析）

### 现象
子女端编辑/删除/新建电子围栏均提示"无权限修改此围栏"，删了重加也不行。

### 根因：云函数权限检查逻辑有缺陷

**`geofence` 云函数的 `updateFence` 检查逻辑**：
```javascript
if (existing.data.creatorId !== creatorId) {
  return { success: false, message: '无权限修改此围栏' };
}
```

**问题链条**：
1. `bind-family` 云函数创建绑定记录在 `family_bindings` 集合，字段名是 `familyId`（=子女 userId）
2. `geofence` 云函数的权限检查查询 `guardians` 集合（可能不存在）+ 回退 `creatorId` 精确匹配
3. 删除重加场景：`creatorId` = 子女端 SharedPreferences 存的 `userId` = `users` 文档 `_id`
4. 如果 `users` 集合中该 deviceId 的文档被修改过（比如字段变更），`_id` 不变，但如果有重复注册逻辑可能返回旧文档

**最可能根因**：
- `family_bindings` 集合存在，字段是 `familyId`
- `guardians` 集合不存在
- 权限检查 fallback 到 `creatorId` 匹配
- 子女端的 `userId` 可能在某次重新注册后变了（ANDROID_ID 不变，但注册逻辑可能查询旧记录）

### 待修复方案
**简化权限检查**：只要是老人的绑定家属（查 `family_bindings` 集合）就能编辑任意围栏，不需要精确匹配 `creatorId`。

修改 `geofence/index.js`：
1. `updateFence` 增加 `elderId` 参数（从请求传入）
2. 权限检查改为：查 `family_bindings` 集合 `{ elderId, familyId: creatorId, status: 'active' }`，找到即有权限
3. 同样修复 `deleteFence`

**子女端也需要改**：在调用 `updateGeofence` 时增加 `elderId` 参数。
