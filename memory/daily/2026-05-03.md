## 14:30 — 核心重构：ImageReader 替代 screencap（v0.41.0）

### 问题
- screencap shell 命令在 MIUI 等国产 ROM 可能被限制
- 50-100ms 延迟，PNG 文件读写浪费 IO

### 解决方案：标准 Android API 管道

```
MediaProjection → VirtualDisplay → ImageReader Surface
     └→ OnImageAvailableListener (RGBA_8888, ~200KB/帧)
          └→ Bitmap → JPEG(30%) → Base64 → HTTP POST 异步上传
```

### 关键改进

| | 旧方案 (screencap) | 新方案 (ImageReader) |
|---|---|---|
| 取帧方式 | 命令行进程 exec | GPU 直接渲染 Surface |
| 延迟 | 50-100ms+ | <5ms |
| 可靠性 | MIUI 可能禁 shell | 标准 API，全厂商支持 |
| 内存 | PNG→Bitmap 双份 | 零拷贝 Buffer→Bitmap |

| v0.40.2 | poll_frame 用 elderId 查文档（之前读错文档导致转圈） |
| v0.41.0 | ImageReader 替代 screencap |

## v0.41.7 — 修复"启动参数异常"

### 问题根因

`startScreenCapture()` 中同时调用了：
```kotlin
startService(serviceIntent)      // → 触发 onStartCommand()
bindService(serviceIntent, ...)  // → 触发 onServiceConnected() → startAfterBind()
```

两个路径都会尝试启动采集，导致冲突。`onStartCommand()` 中的参数检查可能失败，触发 `stopSelf()`。

### 修复

**移除 `startService()`，只用 `bindService()`**：
```kotlin
// 旧代码
startService(serviceIntent)
bindService(serviceIntent, serviceConnection, Context.BIND_AUTO_CREATE)

// 新代码
bindService(serviceIntent, serviceConnection, Context.BIND_AUTO_CREATE)
```

### 流程

```
bindService() → onBind() → onServiceConnected() → startCaptureAfterBind() → startCapture()
```

## 19:07 — v0.41.9 修复 `pendingResultCode` 检查 Bug 🔴

### 根因：Android 常量值的经典陷阱

```kotlin
private var pendingResultCode: Int = -1    // 初始值 = -1
// ...
pendingResultCode = resultCode             // resultCode = Activity.RESULT_OK = -1
```

`Activity.RESULT_OK` 的值就是 `-1`！检查 `pendingResultCode == -1` 永远为 true，无法区分"还没收到 onActivityResult"和"用户授权成功"。

### 修复

```diff
- private var pendingResultCode: Int = -1
+ private var pendingResultCode: Int = Int.MIN_VALUE

- if (pendingResultCode == -1)
+ if (pendingResultCode == Int.MIN_VALUE)
```

`RESULT_OK (-1)` 和初始值 `Int.MIN_VALUE` 不同，检查逻辑恢复正确。

### 教训
- Android 常量值记不住时要查文档：RESULT_OK=-1, RESULT_CANCELED=0, RESULT_FIRST_USER=1
- 用常量值做初始哨兵时必须确认不会和有效值冲突
- 更安全的做法：用 nullable 或独立的 boolean flag

## 19:07 — v0.41.9 修复 `pendingResultCode` 检查 Bug 🔴

### 根因：Android 常量值的经典陷阱

```kotlin
private var pendingResultCode: Int = -1    // 初始值 = -1
// ...
pendingResultCode = resultCode             // resultCode = Activity.RESULT_OK = -1
```

`Activity.RESULT_OK` 的值就是 `-1`！检查 `pendingResultCode == -1` 永远为 true，无法区分"还没收到 onActivityResult"和"用户授权成功"。

### 修复

```diff
- private var pendingResultCode: Int = -1
+ private var pendingResultCode: Int = Int.MIN_VALUE

- if (pendingResultCode == -1)
+ if (pendingResultCode == Int.MIN_VALUE)
```

`RESULT_OK (-1)` 和初始值 `Int.MIN_VALUE` 不同，检查逻辑恢复正确。

### 教训
- Android 常量值记不住时要查文档：RESULT_OK=-1, RESULT_CANCELED=0, RESULT_FIRST_USER=1
- 用常量值做初始哨兵时必须确认不会和有效值冲突
- 更安全的做法：用 nullable 或独立的 boolean flag

## 23:42 — 内存压缩 + 模拟器替代方案调研总结

### 模拟器替代方案调研结论

搜索并分析了所有主流 Android 模拟器在 Mac 上的表现：

| 方案 | 内存要求 | 8GB Mac 适配 |
|------|---------|-------------|
| Android Studio 模拟器（QEMU） | 1.5-3GB | ❌ macOS+OpenClaw 已占 5-6GB，OOM |
| Genymotion（VirtualBox） | 2-4GB+（VB 本身就要 200MB+） | ❌ 比 QEMU 更重 |
| BlueStacks/MuMu | 2GB+ | ❌ 不支持 macOS Intel |
| Firebase Test Lab（云端） | 0（云端跑） | ⚠️ $1/小时，只能 Robo 自动测试，无法手动交互 |
| LambdaTest/BrowserStack | 0 | ⚠️ 主要是 Web 测试，不适合 APK 交互验证 |

**结论：8GB MacBook Air 无轻量替代模拟器。** 所有本地模拟器都需要至少 2GB，8GB Mac 在 macOS+OpenClaw 后剩余不够。

```bash
scrcpy --tcpip=手机IP:5555   # 手机先开 adb tcpip 5555
```
