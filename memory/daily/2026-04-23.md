  ...
}

## 17:58-18:02 - MuJoCo vs 纯 NumPy 合成器可信度对比

### 核心本质区别
| | MuJoCo | NumPy 合成器 |
|--|--------|-------------|
| 原理 | 物理引擎：牛顿力学计算受力/加速度/角速度 | 统计重采样：在已有分布内插值+加噪声 |

| 数据多样性 | 可生成全新场景 | 受限于已有分布范围 |
| 边缘案例 | 能模拟罕见跌倒 | 只能模仿见过的类型 |

### NumPy 合成器的定位
- **不是替代 MuJoCo**，而是在 MuJoCo 已有规律内做**快速插值扩展**
- 当前 features_v2 的 23,177 个样本已是 MuJoCo 生成，NumPy 只补充缺失场景
- 模型学到的物理规律仍然是 MuJoCo 的，NumPy 合成**不损失可信度**
- 天花板：无法生成 MuJoCo 没见过的全新物理模式

### 可信度够用场景
- ✅ 阈值法检测（只看峰值 > 1.5g）

## 20:19 - FallDetector.kt 检测灵敏度分析

### 问题分析
分析 FallDetector.kt 检测流程，发现三步串行检测（冲击→静止→ML推理）过于严格：

| 步骤 | 原阈值 | 问题 |
|------|--------|------|
| 冲击触发 | dynamicAccThreshold × 0.6f (0.6g) | 门槛太高，大量真实跌倒被漏掉 |
| 静止检测 | variance < 0.3, 持续 800ms | 太严格，真实跌倒后不一定"静止" |
| ML 推理 | ML_MIN_PROB ≥ 0.15 | 门槛太高，ML无效时直接跳过 |
| ML 判定 | ML_THRESHOLD ≥ 0.65 | 门槛太高，跌倒概率 >65% 才报警 |

### 根本问题
真实跌倒可能不满足"静止800ms"条件，或冲击波形不满足阈值条件。整个流程串行，任何一步不满足就跳过。

### 修复方案
1. 冲击触发：0.6f → 0.4f（降低40%）
2. 静止检测：800ms → 500ms，variance 0.3 → 0.5
3. ML_MIN_PROB：0.15 → 0.05（让更多ML推理参与判断）
4. ML_THRESHOLD：0.65 → 0.55（更容易判定跌倒）
5. ML无效时：有阈值法兜底（当概率<0.05时检查强冲击 peak>2.0g）

- 原始数据只有理想化重力旋转
- **修复**：新增 `_add_realistic_fall_phases()` 函数

## 22:34 - 🔴 发现 ML 模型推理随机概率根因：Android/Python 特征顺序不匹配

### 核心问题
Android `FallDetectorML.kt` 的 `extractFeatures()` 输出的 77 维特征顺序与 Python `FeatureExtractor` 完全不同。模型在 Python 特征空间上训练，但 Android 端喂的是另一个特征空间的值，导致推理输出接近随机。

### Android 特征顺序（当前错误的）
```
0-4:   acc_mag (mean, std, max, min, range)
5-9:   acc_x (mean, std, max, min, range)
10-14: acc_y (mean, std, max, min, range)
15-19: acc_z (mean, std, max, min, range)
20-39: gyro (mag 5 + xyz 15)
40-43: acc_jerk (mean, std, max, min)
44-45: SMA (acc, gyro)
46-51: skewness/kurtosis (x, y, z)
52-57: correlation (acc_xy, acc_xz, acc_yz, gyro_xy, gyro_xz, gyro_yz)
58-63: energy (acc_x, acc_y, acc_z, gyro_x, gyro_y, gyro_z)
64-65: entropy (acc_mag, gyro_mag)
66-71: FFT peaks (acc 3, gyro 3)
72-75: dominant freq + spectral centroid (acc, gyro)
76:    low band energy (acc_mag)
```

### Python 特征顺序（模型期望的）
```
0-9:   acc_x (mean, std, max, min, range, mad, iqr, skew, kurt, energy)
10-19: acc_y (同上 10 个)
20-29: acc_z (同上 10 个)
30-32: acc_mag (mean, std, max)
33-44: 频域 (每轴: dom_freq, spec_energy, spec_entropy, spec_centroid)
45-52: 姿态 (roll_mean, roll_std, pitch_mean, pitch_std, yaw_mean, yaw_std, roll_change, pitch_change)
53-61: 相关 (acc_gyro_x/y/z, acc_xy/yz/xz, gyro_xy/yz/xz)
62-76: 动态 (jerk_x/y/z mean+max, impact_diff, impact_std_ratio, zero_cross_x/y/z, peaks_x/y/z, sma)
```

### 关键差异
1. **Android 按传感器分组**（acc_mag → acc_xyz → gyro → jerk → ...）
2. **Python 按特征类型分组**（acc_x全10维 → acc_y全10维 → acc_z全10维 → ...）
3. Android 缺少：MAD、IQR、姿态角（roll/pitch/yaw）、impact_diff、impact_std_ratio、zero_cross、peaks
4. Android 多出但位置不对：energy、entropy、gyro_stats 等

### 修复方案
**重写 Android `extractFeatures()` 让它和 Python `FeatureExtractor` 完全一致**

需要新增的 Kotlin 函数：
- `median()` - 中位数
- `mad()` - 中位数绝对偏差
- `iqr()` - 四分位距 (percentile75 - percentile25)
- `percentile()` - 百分位数
- 姿态角计算 (roll, pitch, yaw)
- `impactDiff()` / `impactStdRatio()` - 冲击前后差异
- `zeroCrossingRate()` - 过零率
- `peakCount()` - 峰值计数
- 频域特征: dom_freq, spec_energy, spec_entropy, spec_centroid（每轴独立 FFT）

### 数据单位确认
- Android SensorCollector：加速度已除以 9.8 转为 g 单位 ✅
- Python 训练数据：加速度也是 g 单位 ✅
- 陀螺仪单位：rad/s ✅（Android 和 Python 一致）

### SCALER_MEAN/SCALER_SCALE 问题
- 当前 Android 中的 SCALER_MEAN/SCALE 是基于 Python 特征顺序计算的

## 🔴 ML 推理随机概率根因（22:34 发现）

### 核心问题
Android `extractFeatures()` 输出的 77 维特征顺序与 Python `FeatureExtractor` **完全不同**：
- Android 按传感器分组（acc_mag → acc_xyz → gyro → ...）
- Python 按特征类型分组（acc_x全10维 → acc_y全10维 → ...）
- Android 缺少：MAD、IQR、姿态角、impact_diff、zero_cross、peaks 等
- Android energy 是 mean(x²)，Python 是 sum(x²)

### 文件位置
- Android: `projects/fall-detection-app/app/src/main/java/com/falldetector/diedaobao/ml/FallDetectorML.kt`


## ✅ v0.4.3 - FallDetectorML 特征顺序修复（23:32-23:50）

### 根因
Android `extractFeatures()` 的 77 维特征顺序与 Python 训练时完全不同，模型在 Python 特征空间训练但 Android 喂的是另一个特征空间的值 → 推理概率接近随机。

### 修复内容
1. **重写 `extractFeatures()`** — 特征顺序完全对齐 Python `FeatureExtractor`
2. **新增函数**：`median()`, `mad()`, `percentile()`, `iqr()`, `zeroCrossingRate()`, `peakCount()`, `meanAbsDiff()`, `computeJerkArray()`, `dftMagnitude()`, `spectralEnergy()`, `spectralEntropy()`, `spectralCentroidFreq()`, `dominantFreq()`
3. **修复 `energy()`** — 从 `mean(x²)` 改为 `sum(x²)`（与 Python 一致）
4. **频域特征改为逐轴计算**（之前是 acc_mag 整体，现在 acc_x/y/z 各自 DFT）

### 关键变化
| 特征 | 旧 Android 位置 | 新位置（=Python） |
|------|----------------|------------------|
| acc_x_mean | feat 5 | feat 0 |
| acc_mag_mean | feat 0 | feat 30 |
| 频域能量 | feat 66-76 (mag整体) | feat 33-44 (逐轴) |
| 姿态角 | ❌ 缺失 | feat 45-52 |
| impact_diff | ❌ 缺失 | feat 68 |
| zero_cross | ❌ 缺失 | feat 70-72 |
| peaks | ❌ 缺失 | feat 73-75 |
