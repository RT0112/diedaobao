## 20:08 - MediaProjection 权限录制-回放系统设计

### 背景
- 不同厂商MediaProjection弹窗步骤不同：AOSP 1步、小米2-3步、华为1-2步、OPPO 2步
- 老人无法每次手动操作多步弹窗
- ADB命令方案：用户无法执行ADB命令（没有电脑连接条件）
- 方案最终确认：**录制-回放机制**（用户提出）

### 方案核心
- **首次**：用户手动点击弹窗，App通过AccessibilityService观察并记录点击步骤
- **后续**：App自动按记录的步骤回放点击，自动判断权限是否获取成功
- **判断成功**：SharedPreferences双向通信（mp_granted标志位），AccessibilityService轮询检测
- **失败处理**：单步5秒超时、总流程30秒超时、连续失败3次清除录制数据

### 数据结构
```kotlin
data class PermissionStep(
    val matchText: String,       // 按钮匹配文本
    val matchType: Int,          // 0=CONTAINS, 1=EXACT
    val viewClassName: String?,  // 辅助匹配
    val delayAfterMs: Long       // 点击后等待时间
)
```

### 健壮性设计
| 场景 | 处理 |
|------|------|
| 按钮文本变化 | 模糊匹配(CONTAINS) + className辅助 |
| 系统更新弹窗变化 | 连续失败3次→清除录制→重新录制 |
| 弹窗未出现 | 15秒无窗口变化→超时→提示 |
| 用户中途手动点击 | 停止回放，当作新录制 |
| AccessibilityService未启用 | 检测并引导用户启用 |

### 改动范围
| 文件 | 改动 |
|------|------|
| `RemoteAssistService.kt` | 添加录制/回放逻辑到onAccessibilityEvent |
| `RemoteAssistActivity.kt` | SharedPreferences通信、录制/回放触发 |
| 新增 `PermissionRecordManager.kt` | 封装录制/回放/存储逻辑 |

## 21:39 - v2 修复"录制0步骤"根因（录制-回放系统第三版）

### 根因分析
- **旧方案**（v1）依赖 `VIEW_CLICKED` 事件的 `source` 节点录制步骤

| 节点来源 | `event.source`（系统弹窗经常为null） | `dialogWindow.root`（从弹窗窗口直接获取） |
| 按钮识别 | 无 | **自动识别正向/负向按钮** |
| 匹配方式 | 纯文本 | 文本 + 按钮类型（允许/拒绝/时间选项） |
| 回退策略 | 无 | 匹配不到时找第一个"允许"类按钮 |

## 21:49 - TouchRecordOverlay 修复（TYPE_ACCESSIBILITY_OVERLAY）

### 问题描述
录制0步骤问题持续存在 → v2主动扫描方案仍无法捕获用户点击

### 根因分析
- **旧方案**（TouchRecordOverlay）使用 `TYPE_APPLICATION_OVERLAY`
- `TYPE_APPLICATION_OVERLAY` 层级低于系统权限弹窗（`TYPE_SYSTEM_DIALOG` / `TYPE_SYSTEM_ERROR`）
- Overlay 在弹窗**下面**，触摸事件被弹窗拦截，Overlay 永远收不到触摸

### 修复方案
| 对比 | 修复前 | 修复后 |
|------|--------|--------|
| WindowType | `TYPE_APPLICATION_OVERLAY` | `TYPE_ACCESSIBILITY_OVERLAY` |

### v4 修复方案：SP 驱动

**核心思路**：Activity 只写 SP 标志，Service 的 onAccessibilityEvent 读标志驱动录制。

| 角色 | 旧行为 | 新行为 |
|------|--------|--------|
| Activity | `instance?.startRecordingPermissionSteps()` | 写 SP: `wants_recording=true` |
| Service.onAccessibilityEvent | 不主动检查 | 读 SP `wants_recording` → 自动启动 overlay |
| Activity.onActivityResult | `instance?.stopRecordingPermissionSteps(save=true)` | 写 SP: `mp_granted=true` |
| Service | 不检测 mp_granted | 读 SP `mp_granted=true` → 自动停止录制+保存 |
| SettingsFragment | `instance?.clearRecordedPermissionSteps()` | 直接清 SP `recorded_coords` |

1. **RemoteAssistService.kt** — onAccessibilityEvent 中增加 SP 标志检查逻辑
2. **RemoteAssistActivity.kt** — 所有 `instance?.` 调用改为写 SP
3. **SettingsFragment.kt** — 清除录制改为直接操作 SP
4. **PermissionRecordManager.kt** — 可能需要支持从 Service 端自动触发录制/停止

### 教训（强化）
- **Android Service instance 不可靠**：系统随时回收，instance?. 是定时炸弹
- **跨组件通信必须走持久化**：SP / BroadcastReceiver / Intent，不要用内存引用
- **?. 静默跳过是最危险的 bug**：不报错、不崩溃、只是什么都不做
