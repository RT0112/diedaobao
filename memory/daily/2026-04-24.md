## 20:14 - ML 模型失效根因分析（核心发现）

### 分析过程

用户质疑 v0.9.7 加物理规则补丁是错误方向，要求分析 ML 为什么不准。

**步骤 1：特征提取对齐验证**

对比 Python (FeatureExtractor77) 和 Kotlin (FallDetectorML.kt) 的 80 维特征：
- 时域特征 (0-32)：完全一致（< 1e-8）
- 频域特征 (33-44)：~0.8% 差异（频率分辨率公式差）
- 物理特征 (77-79)：完全一致

**结论：特征提取对齐没问题，问题在训练数据本身。**

---

### 核心发现：SisFall 数据的三个致命问题

**问题 1：传感器位置差异 — 方向性特征在手机上失效**

| 特征 | Cohen d | 说明 |
|------|---------|------|
| `acc_y_mean` | 1.07 | 最强区分特征，但依赖传感器固定方向 |
| `acc_y_energy` | 0.98 | Y轴能量，依赖方向 |
| `roll_mean` | 0.82 | 姿态角，依赖手机朝向 |

- SisFall：传感器固定在腰/胸部 → Y 轴方向一致
- 手机：位置不固定（口袋/手/包）→ Y 轴方向不可预测
- **模型学到的是"Y 轴均值 < -0.6 就是跌倒"，不是物理特征**

**问题 2：加速度峰值区分度极低**

- FALL `acc_mag_max` P50 = 1.41g，ADL P90 = 4.51g
- Cohen d = 0.07，几乎无区分力
- 96.7% 的跌倒样本峰值 < ADL P90
- 模型没学到"峰值 > 3g = 跌倒"

**问题 3：失重特征不可靠**

- 36% 的跌倒样本 `freefall_quality = 0`（无失重期）
- 传感器在腰/胸 ≠ 重心，人体不是完美自由落体
- 缓慢跌倒（滑倒、扶墙倒）几乎无失重

---

### Top 20 特征 Cohen d 排序

```
[10] acc_y_mean          d = 1.07  ← 方向性（手机不可靠）
[19] acc_y_energy        d = 0.98  ← 方向性
[45] roll_mean           d = 0.82  ← 姿态（口袋不可靠）
[39] acc_y_spec_entropy  d = 0.73  ← 方向性
[70] zero_cross_x        d = 0.69  ← 方向性
...
[77] freefall_quality    d = 0.49  ← 物理特征（但 36% 跌倒无失重）
```

**没有一个纯物理的峰值/冲击特征进入 Top 20。**

---

### 结论

**模型学到的是错误的特征：**
- 方向性（`acc_y_mean`）而非物理性（`acc_mag_max`）
- 姿态（`roll_mean`）而非冲击强度

**为什么 ML 在真实数据上失效：**
- 手机位置不固定 → 方向性特征失效
- 真实跌倒冲击 > 3g → 但 SisFall 跌倒 P50=1.41g → 模型没见过

**根本原因：训练数据（SisFall）与真实场景的 domain gap**

---

### 下一步方案

**方案 A：删除方向性特征，重训练纯物理模型**
- 删除：`acc_x/y/z_mean`, `acc_x/y/z_energy`, `roll/pitch/yaw`, `zero_cross_*`
- 保留：`acc_mag_*`, `freefall_quality`, `peak_velocity`, `impact_strength`, `jerk_*` 等
- 重新训练，强制模型学习物理特征

**方案 B：收集真实手机跌倒数据**
- 用手机（口袋位置）捕获跌倒模拟数据
- 包含不同位置变体

**方案 C：增强 SisFall 数据**
- 用物理模拟生成缺失的高冲击跌倒数据
- 添加随机旋转（破坏方向性特征的可靠性）

## 22:27 - v10 训练/推理分布不匹配诊断 & v11 完整方案设计

### 额外发现：v10 训练数据与推理数据严重不匹配

用户反馈：
- "摔好多次都不报" — 真实跌倒大量漏报
- "只有手机自由落体放下去的时候才会报" — 只有手机自由落体才触发

**根本原因分析**：

| 维度 | v10 训练数据 | Android 推理 |
|------|-------------|-------------|
| 数据来源 | `sisfall_physics_params`（文件级物理参数） | 2.5s 滑动窗口 |
| acc_peak_g | 全文件最大值（FALL mean=**7.27g**） | 窗口内最大值 |
| 样本数 | 1847（900 FALL + 947 ADL） | 每秒 1 次推理 |
| 峰值捕获 | 保证捕获最大冲击 | 可能错过峰值时刻 |

**验证数据**：
- `extract_physics_peak_aligned.py` 测试了滑动窗口提取：
  - SisFall 第一个跌倒文件：file_max = **7.10g**
  - 2.5s 窗口的 window_max = 7.10g（峰值窗口）
  - **但 window_p50 = 0.98-1.83g**（大部分窗口不包含峰值）

**结论**：v10 模型学会了"峰值 > 5g = 跌倒"，但真实推理时大部分窗口的峰值只有 1-2g。

| TYPE_LINEAR_ACCELERATION | ❌ 未用 | 去除重力后的纯运动信号 |
| TYPE_ROTATION_VECTOR | ❌ 未用 | 姿态变化检测 |

### v11 完整方案设计

**核心改进**：
1. 重构训练数据：从 SisFall 原始数据用滑动窗口提取特征，保证训练/推理分布一致
2. 多传感器融合：充分利用重力传感器（gravity-relative 特征）、陀螺仪等
3. 特征扩展：6 维 → 22 维混合特征

**22 维特征设计**：

**A 组：加速度 magnitude（8 维，旋转不变）**
- acc_peak_g, acc_mean_g, acc_std_g, acc_min_g

**B 组：重力相对方向特征（6 维，位置无关）**
> 关键创新：重力始终朝下，不管手机怎么放，重力方向不变

**训练数据重建流程**：
```
SisFall 原始数据 → 峰值检测 → 峰值对齐窗口提取
              → FALL 窗口（峰值 ± 1.25s）= 正样本
              → ADL 窗口（随机采样）= 负样本

```

**预期效果**：
- 真实跌倒检测率：10% → **> 85%**
- FNR：< 5%
- FPR：保持低位
