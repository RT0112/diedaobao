## 跌倒宝 Android 模拟器测试

### 结论：8GB MacBook Air 无法运行 Android 模拟器

**问题根因**：模拟器启动过程中崩溃，launcher 本身因内存不足退出，qemu 子进程未能启动。

**实测过程**：
1. 关闭所有 App 后尝试启动，仍然崩溃
2. 模拟器 launcher 开始执行，输出正常启动日志
3. 到达 "Saving snapshot 'default_boot'" 阶段后退出
4. 监控显示 qemu-system 进程从未出现（0KB 内存占用）

**日志关键输出**：
```
Increasing RAM size to 1536MB  # 实际被 AVD 配置覆盖
Saving snapshot 'default_boot' using 1 ms
(emulator launcher then exits)
```

**原因分析**：
- MacBook Air 只有 8GB RAM
- Android 模拟器 boot 阶段需要大量内存
- macOS 内存不足时，进程被内核 OOM Killer 终止

### 推荐方案：scrcpy + 真机测试

```bash
# Mac 上装 scrcpy
brew install scrcpy

# 红米 K90 (Android 14/15) 通过 USB 连接
# 开启「开发者选项 → USB 调试」
scrcpy  # 无线投屏到 Mac
```

**优点**：Mac 只负责显示，几乎零内存占用

## v0.43.1 — VirtualDisplay MIUI Flag 兼容修复

### 背景
v0.43.0 诊断版发现 VirtualDisplay 在 MIUI/HyperOS 上创建失败（返回 null），原因是 `VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` 在 MIUI 上不兼容。

### 根因
`VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` 在小米/红米 MIUI/HyperOS 系统上直接返回 null，无法创建 VirtualDisplay。

### 修复方案：三组 Flag 自动重试
按兼容性从高到低依次尝试：
1. `OWN_CONTENT_ONLY | PUBLIC`（0x19）— MIUI 兼容首选
2. `PUBLIC` 单独 — 次选
3. `AUTO_MIRROR` — 兜底

同时将分辨率限制为 720x1280，减少 MIUI 内存压力。

### 通知栏预期输出
```
创建VirtualDisplay 720x1280...
尝试VD flag组合1: 0x19
VD创建成功: flag组合1
✅ 推流已启动, 等待首帧...
```

### APK 交付
- **版本**: v0.43.1 (versionCode=129)
- **APK**: `app/build/outputs/apk/debug/`

### 待用户操作
安装到红米 K90，启动远程协助，观察通知栏显示哪组 flag 成功。

### 关键教训
- MIUI/HyperOS 对 `VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` 支持差，应优先使用 `OWN_CONTENT_ONLY | PUBLIC`
- VirtualDisplay flag 需要多组降级重试，不能只依赖单一 flag
- 分辨率越高 MIUI 内存压力越大，720p 是较安全的默认值

## MIUI 远程协助踩坑总结

### VirtualDisplay 在 MIUI 上不可靠
| 方案 | 结果 |
|------|------|
| `VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` | MIUI 返回 null ❌ |
| `OWN_CONTENT_ONLY \| PUBLIC` (0x19) | MIUI 返回 null ❌ |
| `PUBLIC` 单独 | MIUI 返回 null ❌ |

**结论**：MIUI/HyperOS 完全阻止了 `createVirtualDisplay()`，flag 重试无效。

### 可靠兜底方案：screencap shell 命令
- `Runtime.getRuntime().exec(arrayOf("screencap", "-p", path))` 在所有 Android 设备上可用
- 缺点：帧率低（1-2fps）、延迟高、需要写文件再读取
- 优点：不依赖 MediaProjection/VirtualDisplay，MIUI 无法阻止

### 最佳实践
1. **启动前检查悬浮窗权限**：`Settings.canDrawOverlays(this)`，MIUI 可能要求此权限才允许录屏
2. **VD 失败自动降级**：不要假设 VD 一定成功，必须有 screencap 兜底
3. **通知栏实时反馈**：用户无 logcat 时，通知栏是唯一诊断手段

## v0.43.1 — VirtualDisplay MIUI Flag 兼容修复

### 背景
v0.43.0 诊断版发现 VirtualDisplay 在 MIUI/HyperOS 上创建失败（返回 null），原因是 `VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` 在 MIUI 上不兼容。

### 根因
`VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` 在小米/红米 MIUI/HyperOS 系统上直接返回 null，无法创建 VirtualDisplay。

### 修复方案：三组 Flag 自动重试
按兼容性从高到低依次尝试：
1. `OWN_CONTENT_ONLY | PUBLIC`（0x19）— MIUI 兼容首选
2. `PUBLIC` 单独 — 次选
3. `AUTO_MIRROR` — 兜底

同时将分辨率限制为 720x1280，减少 MIUI 内存压力。

### 通知栏预期输出
```
创建VirtualDisplay 720x1280...
尝试VD flag组合1: 0x19
VD创建成功: flag组合1
✅ 推流已启动, 等待首帧...
```

### APK 交付
- **版本**: v0.43.1 (versionCode=129)
- **APK**: `app/build/outputs/apk/debug/`

### 待用户操作
安装到红米 K90，启动远程协助，观察通知栏显示哪组 flag 成功。

### 关键教训
- MIUI/HyperOS 对 `VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` 支持差，应优先使用 `OWN_CONTENT_ONLY | PUBLIC`
- VirtualDisplay flag 需要多组降级重试，不能只依赖单一 flag
- 分辨率越高 MIUI 内存压力越大，720p 是较安全的默认值

## MIUI 远程协助踩坑总结

### VirtualDisplay 在 MIUI 上不可靠
| 方案 | 结果 |
|------|------|
| `VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` | MIUI 返回 null ❌ |
| `OWN_CONTENT_ONLY \| PUBLIC` (0x19) | MIUI 返回 null ❌ |
| `PUBLIC` 单独 | MIUI 返回 null ❌ |

**结论**：MIUI/HyperOS 完全阻止了 `createVirtualDisplay()`，flag 重试无效。

### 可靠兜底方案：screencap shell 命令
- `Runtime.getRuntime().exec(arrayOf("screencap", "-p", path))` 在所有 Android 设备上可用
- 缺点：帧率低（1-2fps）、延迟高、需要写文件再读取
- 优点：不依赖 MediaProjection/VirtualDisplay，MIUI 无法阻止

### 最佳实践
1. **启动前检查悬浮窗权限**：`Settings.canDrawOverlays(this)`，MIUI 可能要求此权限才允许录屏
2. **VD 失败自动降级**：不要假设 VD 一定成功，必须有 screencap 兜底
3. **通知栏实时反馈**：用户无 logcat 时，通知栏是唯一诊断手段

## 远程协助 elderId 数据流深度分析（10:56）

### 问题：子女端依旧转圈，看不到老人画面

### 完整数据流追踪

**老人端上传帧**：
```
ScreenCaptureService.onImageAvailable()
  → elderId = intent.getStringExtra(EXTRA_ELDER_ID)
  → HTTP POST /remote-assist {action:"upload_frame", userId: elderId, data: base64, ...}
  → 云函数写入 elderId 文档的 frameBuffer
```

**elderId 来源**：
```
RemoteAssistActivity.startScreenCapture()
  → val elderId = getBoundElderId() ?: getUserId() ?: ""
  → getBoundElderId() = SharedPreferences("cloudbase").getString("elder_id", null)
  → getUserId() = SharedPreferences("cloudbase").getString("user_id", null)
```

**子女端拉帧**：
```
RemoteAssistManager.pollFrame()
  → HTTP POST /remote-assist {action:"poll_frame", elderId: boundElderId, lastFrameNum: N}
  → 云函数: targetId = elderId || userId → 查 targetId 文档 → 返回 frameBuffer
```

**子女端 elderId 来源**：
```
RemoteAssistManager.initialize(guardianUserId, boundElderId)
  → 需要传入正确的 boundElderId
```

### 关键风险点

1. **老人端 `elder_id` 是否写入？**
   - CloudBaseClient.kt 有 `saveElderIdToPrefs()` 方法
   - 在 `registerUser()` 时调用
   - 但如果用户已经注册过（旧版本），可能没有 `elder_id` 键
   - 此时 fallback 到 `user_id` 键

2. **子女端 `boundElderId` 是否正确传入？**
   - 需要确认子女端 Activity 初始化 RemoteAssistManager 时传入的值
   - 如果子女端 SharedPreferences 中的 `elder_id` 与老人端 `user_id` 不一致 → 数据错位

3. **云函数 pollFrame 逻辑**：
   - `targetId = elderId || userId`
   - 子女端传了 `elderId` → `targetId = elderId`
   - 如果这个 elderId 与老人端上传时的 userId 不一致 → 永远拉不到帧

### 创建的诊断工具
- `diagnose_remote_assist.py`：Python 脚本，可检查云函数连通性、帧 buffer 状态
- 用法：`python3 diagnose_remote_assist.py <elderId>`

### 下一步
需要用户提供老人端的实际 elderId/userId，才能用 curl/脚本验证云端是否有帧数据。

## v0.43.1 — VirtualDisplay MIUI Flag 兼容修复

### 根因

### 修复方案：三组 Flag 自动重试

1. `OWN_CONTENT_ONLY | PUBLIC`（0x19）— MIUI 兼容首选

3. `AUTO_MIRROR` — 兜底

同时将分辨率限制为 720x1280，减少 MIUI 内存压力。

尝试VD flag组合1: 0x19
VD创建成功: flag组合1
✅ 推流已启动, 等待首帧...

## MIUI 远程协助踩坑总结

### VirtualDisplay 在 MIUI 上不可靠
| 方案 | 结果 |
|------|------|
| `VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` | MIUI 返回 null ❌ |
| `OWN_CONTENT_ONLY \| PUBLIC` (0x19) | MIUI 返回 null ❌ |
| `PUBLIC` 单独 | MIUI 返回 null ❌ |

**结论**：MIUI/HyperOS 完全阻止了 `createVirtualDisplay()`，flag 重试无效。

### 可靠兜底方案：screencap shell 命令
- `Runtime.getRuntime().exec(arrayOf("screencap", "-p", path))` 在所有 Android 设备上可用
- 缺点：帧率低（1-2fps）、延迟高、需要写文件再读取
- 优点：不依赖 MediaProjection/VirtualDisplay，MIUI 无法阻止

### 最佳实践

## MIUI VirtualDisplay 不可靠总结

| 方案 | 结果 |
|------|------|
| `VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR` | MIUI 返回 null ❌ |
| `OWN_CONTENT_ONLY \| PUBLIC` (0x19) | MIUI 返回 null ❌ |
| `PUBLIC` 单独 | MIUI 返回 null ❌ |

**结论**：MIUI/HyperOS 完全阻止 `createVirtualDisplay()`，flag 重试无效，必须 screencap 兜底。

## 经验教训

### CloudBase 数据是最可靠诊断来源
- 查询数据库比问用户更准确
- `family_bindings` 集合中 `familyId = elderId` 说明自己绑自己
- `users` 表中 role 字段验证用户类型

### Android 陷阱
- **ANDROID_ID 是设备级唯一**：同设备多 App 共享，必须加前缀区分
- **SharedPreferences 文件名是隐式接口**：拼错不报错但数据隔离
- **异步+suspend 混用是隐蔽竞态 bug**：syncBindingFromCloud 完成前 getElderId 返回空

### 网络与超时
- HTTP 超时留足 30 秒（CloudBase API 弱网响应慢）
- AppLogger 能告诉我们用户是否注册（userId="not_registered_yet"）
