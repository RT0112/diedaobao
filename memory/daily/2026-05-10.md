## 00:02 RemoteAssistService 录制流程修复

### Bug: TouchRecordOverlay 未激活导致录制为空
- **根因**: 用户点"好的，开始"后直接调 `requestMediaProjection()` 弹系统弹窗，但没有先启动 MediaProjection 录制。TouchRecordOverlay 没激活，弹窗上用户的触摸操作完全没被拦截录制。
- **修复**: 在 `requestMediaProjection()` 之前先调 `RemoteAssistService.instance?.startMpRecording()`，激活 TouchRecordOverlay 拦截触摸坐标。

### RemoteAssistService API 注意事项
- `getInstance()` 是错误的调用方式，正确的是 `instance` 属性（Kotlin object 单例的 @JvmStatic companion 属性）
- 用 `RemoteAssistService.instance?.xxx()` 而非 `RemoteAssistService.getInstance()?.xxx()`

### 修复后的完整流程
1. 点"允许" → 录制引导弹窗
2. 点"好的，开始" → **先启动录制(Overlay激活)** → 再弹出 MediaProjection 系统弹窗
3. 用户操作弹窗 → Overlay 拦截每次触摸坐标 → 录进 recordedSteps
4. 授权成功 → `onActivityResult` → 保存录制

  ```kotlin
      .apply()
  ```

## 00:30 录制坐标误录修复 + 取消自动禁用

### Bug: 录到的坐标全是 App UI 点击，不是弹窗按钮
- **根因**: TouchRecordOverlay 在 `requestMediaProjection()` 之前就开始录制，用户点"好的，开始"按钮的触摸被录进去（step 0: (1085, 2401) 是"好的，开始"按钮坐标）
- **K70 屏幕分辨率**: 1440×3200，录到的坐标全在下半屏（App UI 区域）
- **连锁问题**: 回放失败3次 → 自动禁用 → 重置又被覆盖

### Fix 1: 录制忽略前1秒触摸
- `recordTouchCoord()` 加 1000ms 冷启动期，`System.currentTimeMillis() - recordingStartTime < 1000` 时直接 return
- 跳过"好的，开始"按钮的误录

### Fix 2: 取消自动禁用
- 删除 `PermissionRecordManager` 中的自动禁用逻辑（回放失败不再锁死，允许无限重试）
- `tryAutoHandle()` 失败只 log 不禁用
- `startPermissionFlow()` 不再检查 `isAutoDisabled`
- `saveRecording()` 不再清除 KEY_AUTO_DISABLED（因为不再有禁用逻辑）

## 15:00 v17.2 修复：回放死循环（手机失控点点点不停）

### Bug 1: 远程协助结束后，下次请求迟迟不来或不弹窗 + 自动授权回放失效

#### 根因分析（代码级）

**根因1：`onDestroy()` 发过期的 `respondToRequest(false)` 拒绝新请求**
- `RemoteAssistActivity` 是 `singleTask`，第一次会话结束 → `finish()` → `onDestroy()`

- 问题：如果云端已经有第二个请求排队，这个 `false` 会**拒绝新请求**
- `scope` 是 `CoroutineScope(Dispatchers.IO + SupervisorJob())`，协程可能比 Activity 生命周期长

**根因2：`handleNewRequest()` 不清理旧会话**
- 只调 `RemoteAssistManager.stopSignalPolling()`，没有：

- 第一次会话 `onDestroy()` 注销了 `autoPermissionReceiver`
- 如果 `singleTask` Activity 的第二次请求走 `onNewIntent()`（Activity 仍在），接收器还活着
- 但如果 Activity 已 `finish()` 后重建走 `onCreate()`，接收器重新注册，OK
- **关键问题**：走 `onNewIntent()` 时 `handleNewRequest()` 没有重新注册广播接收器

## 21:28 v19.6 找到自动断开真正根因 + 修复

### 真正的根因
`handleNewRequest()` 无条件调 `scope.launch { RemoteAssistManager.endAssist() }` 来"清理旧会话"，但 `endAssist` 是异步的：

```
第1次请求 → 没有 isAssisting → 不调 endAssist → 正常 ✅
第2次请求 → 调 endAssist(异步) → 用户点允许 → 云端 status=active → endAssist 才执行 → status 改回 idle → checkAssistStatus 检测到 idle → 断开！💥
```

这就解释了为什么"第一次不自动断，后面几次请求的就又自动断了"——第一次没有旧会话，endAssist 无害；第二次异步 endAssist 在新会话建立后才执行，把新会话也杀了。

### 修复
1. `handleNewRequest` 只在 `isAssisting=true` 时才调 `endAssist`（确实有旧会话才清理）
2. `checkAssistStatus` 重新启用，加**双重确认**：连续2次查到 `idle` 才断开（防偶发时序问题）
3. 非 idle 状态重置确认计数，网络异常不断开

### 排查过程回顾
- v19.3: 回退云函数推信号方案，只用 checkAssistStatus → 自动断
- v19.4: 只对 idle/cancelled 断开，unknown 忽略 → 还是第二次后断
- v19.5: 完全禁用 checkAssistStatus → 不自动断了，但子女端断开老人端不知道
- v19.6: 找到根因是 handleNewRequest 异步 endAssist 杀新会话

## 21:38 v19.7 彻底换方案：信号推模式替代状态查询

## 23:50 子女端帧轮询超时+请求被拦截修复

### Bug根因
1. **帧轮询永不超时**: `emptyCount` 每30次归零，空帧轮询永远不断开，状态一直是 CONNECTING/STREAMING
2. **新请求被拦截**: `requestAssist` 检查 `currentState != IDLE` 直接 return，旧会话残留时永远无法发新请求
3. **表现**: 子女端点请求 → "其他家属正在协助中"（其实云端是idle，是App内部状态残留）

### 修复内容 (family-guardian-app RemoteAssistManager.kt)
1. **帧轮询60s超时**: 新增 `MAX_EMPTY_COUNT = 200`（200×300ms=60s），累计空帧达200次自动断开为 DISCONNECTED
2. **requestAssist自动清理**: 状态非IDLE时先调 `stopPolling()` + 重置为 IDLE，再发新请求
3. **stopPolling增加空帧计数重置**: `emptyCount = 0` 清零

### 教训
- 任何轮询/循环必须有超时退出机制，不能只靠"状态检查"
- 请求入口必须处理"旧会话残留"场景，不能假设状态一定干净
- emptyCount 归零逻辑不能是无条件循环归零，必须是累计计数+超时阈值
