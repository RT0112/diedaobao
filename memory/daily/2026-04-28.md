## 下午：FF合并Bug分析（15:55）

### 发现的两个Bug

#### Bug1：兜底逻辑用错变量
`detectVelocityTrend` 的 history.size < 4 兜底判断用的是类变量 `freefallTimeMs`（上一段FF的时长），而非当前段时长。
导致：如果上一段FF >= 100ms，即使当前段 < 100ms 也会错误合并。

#### Bug2：速度累积值段间污染
上下运动时，velocity 是积分累积值（带方向）：
- 上：velocity 正向累积 (+1, +2, +3)
- 间隔期：加速度反向，velocity 减少
- 下：velocity 继续变负 (-2, -1, 0, +1, +2)
合并后只有"下"的累积值，方向递增 → 被误判为趋势递增 → 合并 ✅（错误！）

### 待办
- [ ] 修复 Bug1：兜底逻辑用当前段时长而非类变量
- [ ] 修复 Bug2：速度趋势判断前检查整体方向是否有大转折
- [ ] git commit 修复版

## 下午续：FF合并Bug深度分析（16:08）

### 关于 Bug1 的更正

之前说 `freefallTimeMs` 是"上一段"的残留是错误的。更正：
- `freefallTimeMs` 在每次冲击触发时从 `process500msWindow()` 结果中提取（取 max FF duration）
- 它是当前这次冲击对应的 500ms 窗口内的 FF 时长，值是对的
- 语义上应该用 section 自己的 duration，但实际值不影响正确性

### 真正的 Bug 根因

**速度趋势用全局累积 signedVelocity，无法区分段间方向**

场景：快速上下甩手机
- 上阶段: accMag<0.7g → velocity 正向累积 (+0.5, +1.0, +1.5, +2.0)
- 间隔期: accMag≈1.0g → 加速度接近零，velocity 几乎不变
- 下阶段: accMag<0.7g → velocity 继续正向累积（因为上阶段累积值高）
- 合并后 allVelocities = 全程正向递增 → 误合并

关键：间隔期没有明显的减速超重（accMag 只在 1.0g 左右，不是 >1.5g），速度方向没有反转。

### 待修复方案

在合并判断前，加方向反转检测：
- 比较段1最后 velocity 和段2第一 velocity 的符号是否反转
- 如果符号从正变负或从负变正，不合并
- 同时可检查加速度方向变化（通过段内平均 accMag 与 1.0g 的关系）

## 22:57 息屏操作问题

### 用户提问
无障碍服务在屏幕关闭时能否操作微信？

### 技术答案
**不能** - AccessibilityService 可以读取屏幕内容（息屏也能读），但模拟点击需要 UI 渲染，屏幕必须亮。

### 解决方案
用 PowerManagerWakeLock 先唤醒屏幕再操作：
```kotlin
val wakeLock = powerManager.newWakeLock(
    PowerManager.SCREEN_BRIGHT_WAKE_LOCK,
    "MyApp:WeChatWakeLock"
)
wakeLock.acquire(10000) // 亮10秒
// 操作微信
wakeLock.release()
```

### 决定
采用方案3：检查屏幕状态，动态决定。点亮屏幕1秒左右对用户体验影响很小。
