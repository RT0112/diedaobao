## 2026-04-26 20:45 v0.24算法完整逻辑文档

### 核心状态机（两状态）
```
MONITORING → IMPACT_DETECTED → (决策链) → MONITORING
```

### 三条决策路径
1. **路径1**：ML >= 0.80 → 直接报警
2. **路径2**：ML >= 0.50 → 加权评分 = 物理分×36% + ML×64% ≥ 0.50
3. **路径3**：ML < 0.50 → 物理覆盖（物理≥80% + 冲击≥3g + FF≥200ms + ML≥25%）

|------|-----|

| 确认期 | 1500ms |
| 取消方差 | 0.5 |
| ML高阈值 | 0.80 |
| ML低阈值 | 0.50 |
| 冷却期 | 2800ms |

### v0.24重设计删除项
- ❌ FALL_CONFIRMED状态（合并到IMPACT_DETECTED）

```

速度评分 = min(峰值速度/3.0m/s, 1.0)
物理总分 = FF×35% + 冲击×35% + 速度×30%
```

```kotlin
recentAccMag[0..49] // 最近1秒的accMag历史

    // 冲击前1秒内有超重 → 排除（向上推时手在加速产生超重）

```
**物理原理**：向上推手机时手在加速，产生超重（accMag>1.2g）；真跌倒FF期间失重（accMag<0.7g），不可能超重。

#### 2. FF合并时超重检测
```kotlin
if (currentGap > 0 && currentGap <= 500 && gapMaxAccMag <= 1.2g) {

```
**物理原理**：正常跌倒FF之间是腿部支撑（accMag≈1g），不会超重；向上推/停顿会产生超重，阻止合并。

或在合并逻辑处，在最终确认 FF 时间后设置。

## 2026-04-26 22:10 向上推手机误报根因深入分析

### 新发现：多段FF合并绕过防护（合并阈值1.0g不够用）

测试场景（`test_push_merge.py`）结果：

| 场景 | FF合并 | 冲击 | 结果 | 正确？ |
|------|--------|------|------|--------|

| Y 推手机合并+强冲击 | 400ms | 3.5g | 🚨 跌倒 | ❌ **误报！** |

| W 推手机不合并 | 80ms | 2.8g | ❌ 排除 | ✅ |

**根因**：推手机时手抓握手机的间隔期 accMag ≈ 0.85-1.0g（< 1.0g 超重阈值），被判定"无超重"→ 合并 → 3段80ms FF合并成400ms → 绕过200ms排除规则 → 误报。

### 推手机 vs 真跌倒的物理特征对比

| 特征 | 推手机（手抓握） | 真跌倒（腿部支撑） |
|------|------------------|---------------------|
| 间隔期 accMag | 0.85-1.0g | 0.9-1.1g |
| 间隔期时长 | 80-200ms | 100-500ms |
| 每段 FF 时长 | 短（<100ms） | 较长（100-300ms） |
| FF段数量 | 多（3-5段） | 少（2-3段） |
| 总事件时长 | 短（<1秒） | 长（1-2秒） |
| 冲击峰值 | 中（2-4g） | 高（3-6g） |

### 洞察：仅靠 accMag 无法区分

推手机间隔期 accMag (0.85-1.0g) 与腿部支撑间隔期 accMag (0.9-1.1g) 区间重叠，单靠 accMag 阈值无法可靠区分。

### 建议方案：平均FF段时长规则

- 合并后如果每段平均 FF 时长 < 100ms → 不是真跌倒（推手机或其他假动作）
- 真跌倒每段 FF 较长（100-300ms）
- 实现：追踪 FF 段数和总时长 → `avgFFDuration = totalFFTime / segmentCount`

## 2026-04-26 22:30 v0.25.4 修复推手机误报（手推超重在FF前）

### 根因分析

推手机误报的真正原因：**手推超重发生在FF段开始之前**。

- 推手机：站立 → 手推产生超重(1.15-1.80g) → 松手 → **FF开始** → 冲击
  - `pauseMaxAccMag` 在FF**之后**才统计，FF开始时pause已被清零
  - 合并时 `pauseMaxAccMag = 0`（因为超重在FF开始前就发生了）
- 真跌倒：站立 → 失重(FF) → 腿部支撑(正常g) → 冲击

**关键问题**：手推产生的超重在FF段开始之前，此时pause尚未激活。合并逻辑检测的是FF间隔期的超重，但推手机的手推超重在FF开始前就发生了，pause被重置为0 → 被判定"无超重" → 合并FF → FF累计≥200ms → 绕过排除 → 误报。

### 修复方案

新增两个追踪器，在FF链开始前检测站立期的超重：

```kotlin
standingMaxAccMag  // 站立期滚动最大值（≥1.05g = 手推超重）
preFfOverweight    // 本次FF链是否曾有前置超重
```

FF链第一段开始时：
```
if (standingMaxAccMag > 1.05g) preFfOverweight = true
```

合并时：
```
if (preFfOverweight == true) 禁止合并  // 不允许"手推超重"事件的FF与后续合并
```

阈值1.05g：过滤站立期传感器噪声(±0.02g)，真实推手机的超重≥1.15g可触发。

### 修复代码

1. `standingMaxAccMag` 追踪器（monitoring状态，每帧更新）
2. `preFfOverweight` 标记（FF链第一段开始时设置）
3. `shouldMergeFF` 新增条件：`preFfOverweight == false`
4. `resetPhysicsState` 重置两个变量

## 2026-04-26 23:36 v0.25.5 编译 - 滚动窗口修复

### 根本问题定位

v0.25.4 的单一变量 `standingMaxAccMag` 存在致命Bug：
- 它只在 `freefallStart==0 && pauseStart==0` 时更新
- 推手机的推力发生在 FF **已经开始之后**（此时 freefallStart≠0）→ 变量不更新
- 单一变量只升不降，上次事件残留污染下次

### v0.25.5 解决方案

用滚动窗口 `standingAccMagWindow`（LinkedList<Pair<accMag, timestamp)>，保留最近500ms）替代单一变量：

```
每帧无条件: standingAccMagWindow.add(accMag, now)
            清理超窗口数据
preFFWindowMax = standingAccMagWindow.maxOf(accMag)  ← 每帧计算

FF链第一段开始时:
  if (preFFWindowMax > 1.05f) preFfOverweight = true
  → 禁止FF合并
```

Python模拟器验证了所有关键场景：
- 推手机（→正确标记preFfOverweight→阻止FF合并→减少误报）
- 缓慢放下（→无FF→不受影响）
- 口袋走路（→FF<200ms→直接排除→不受影响）
- FF<200ms→不受影响（窗口最大可能约1.05g，不触发阈值）
