## 09:00 - K70独立穿透突破（不依赖Mac）

### 核心突破：修复 Termux 下 Go 程序 DNS+TLS 双重问题

**根因分析：**
1. **DNS失败** — Go 程序找 `/etc/resolv.conf`，Termux 没有此文件（在 `$PREFIX/etc/resolv.conf`），Go fallback 到 `[::1]:53`（IPv6 localhost）→ 连接被拒
2. **TLS失败** — Go 程序找 `/etc/ssl/certs/`，Termux 的 CA 证书在 `$PREFIX/etc/tls/` → x509 证书验证失败
3. 之前误判为"Termux DNS 兼容问题无法解决"，实际是路径映射问题

**解决方案：proot 绑定**
```bash
SSL_CERT_FILE=$PREFIX/etc/tls/cert.pem proot \
    -b $PREFIX/etc/resolv.conf:/etc/resolv.conf \
    -b $PREFIX/etc/tls:/etc/ssl \
    ~/cloudflared tunnel --url http://localhost:3000 --protocol http2
```
- `proot -b` 把 Termux 路径绑定到 Go 程序期望的标准 Linux 路径
- `SSL_CERT_FILE` 环境变量指定 CA 证书位置
- `--protocol http2` 避免 QUIC 在 Termux 下的兼容问题（QUIC 会 Error 1033）

**关键经验：Android Termux 下 Go 程序网络问题的通用修复模式**
- proot 绑定 resolv.conf + ssl 目录 + SSL_CERT_FILE 环境变量
- 适用于所有 Go 编译的 CLI 工具（ngrok/cloudflared/frp 等）

## 11:45 - 4个Bug修复第二轮（WS+HTTP双重触发根因分析）

### 根因发现：WS和HTTP轮询双重触发协助请求

**日志证据（两次 onNewIntent）：**
- `11:37:09` — `[WS] 收到协助请求: from=测试家属` → WS推送的请求
- `11:37:11` — `收到协助请求: from=测试家属，通知 2 个监听器` → HTTP轮询的请求

**根因：** `startPolling()` 中 `WSClient.connect()` 发起连接后立即检查 `isWSConnected()`，此时WS还没连上返回false，于是启动HTTP轮询。WS连上后，HTTP轮询不会自动停止，导致WS和HTTP同时推送请求。
