### 🔴 flushAuditLog 竞态覆盖 Bug（已定位，修复中）

**问题**：`flushAuditLog` 使用整体覆盖写回 `remoteAssist`，导致并发时 frameBuffer 丢失

**根因**：
```js
// 旧代码（危险）：
const ra = (res.data[0].remoteAssist || {})
const trail = (ra.auditTrail || []).slice(-150)
trail.push(..._auditEntries)
await db.collection(USERS_COL).doc(targetUserId).update({
  remoteAssist: { ...ra, auditTrail: trail }  // ⚠️ 覆盖整个 remoteAssist！
})
```

当 `flushAuditLog` 读 `ra` 和写回之间，如果 `uploadFrame` 已 push 新帧到 frameBuffer，
flush 写回时就会用旧 ra 覆盖，**新帧数据丢失**。

**修复方案**：改用 dot notation 只更新 `auditTrail`，不覆盖整个 `remoteAssist`：
```js
// 新代码（安全）：
await db.collection(USERS_COL).doc(targetUserId).update({
  'remoteAssist.auditTrail': trail  // ✅ 只更新 auditTrail，不碰 frameBuffer
})
```

**状态**：分析完成，修复代码需写入文件并部署

**竞态分析总结**：
- `flushAuditLog` 用 dot notation 更新 `auditTrail` ✅
- `updateRemoteAssist` 用 dot notation 更新各字段 ✅
- 两者互不干扰，不再有覆盖风险
- `upload_frame` / `poll_frame` 不触发 flush（`auditTarget` 为 null），安全
- `upload_frame` 内部的 read-modify-write 对 frameBuffer 仍有轻微竞态（偶尔丢一帧），但可接受

### 关键教训
- **RGBA_8888 vs ARGB_8888**：Android Bitmap 内部是 ARGB_8888，但 `ImageReader` 的 `Image.Plane` buffer 字节序在 RGBA_8888 格式下实际就是 R,G,B,A 逐字节排列，与 ARGB int 的字节序一致，不需要做 R↔B swap。之前误以为是 BGRA 导致颜色反转。
- **通知渠道创建失败 = fullScreenIntent 不工作**：`DEFAULT_VIBRATE_SETTINGS` 是 AudioAttributes 常量不是 VibrationEffect，用它设置 vibrationPattern 编译不过；渠道创建异常被吞掉后 fullScreenIntent 无法触发，导致只显示普通通知不覆盖界面。
- **cleanup 逻辑要统一**：屏幕共享断开时分散的 cleanup 容易遗漏步骤（解绑、停服务、取消通知、停轮询），必须用一个统一函数处理。
