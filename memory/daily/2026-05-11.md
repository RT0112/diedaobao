## 09:30 子女端远程协助切Tab画面丢失修复

### Bug描述
子女端远程协助连接中（STREAMING状态），切换底部导航栏到首页再切回远程协助，画面消失，回到发起请求的初始界面。

### 根因
Navigation + BottomNavigationView 架构下，切 tab 时 Fragment 走 `onDestroyView` → `onCreateView` → `onViewCreated` 生命周期。`onDestroyView()` 调了 `manager.dispose()`，把帧轮询协程杀掉 + 状态重置为 IDLE。切回来时新 View 重建，但 manager 已死，无法恢复。

### 修复
- `manager` 改为 `by lazy` 初始化，Fragment 级别保留（不随 View 重建）
- `onDestroyView` 只清理 UI 资源（stopTimer、回调置 null），不调 dispose
- `onViewCreated` 加 `restoreUIFromState()`，根据 currentState 恢复对应 UI 容器
- STREAMING 状态切回来：自动恢复计时器 + 帧回调重新绑定到新 ImageView
- `dispose()` 只在 `onDestroy` 调用（Fragment 真正销毁时）
- `RemoteAssistManager.currentState` 改为可读属性，方便 Fragment 查询

### 修改文件
- `RemoteAssistFragment.kt`
- `RemoteAssistManager.kt`

### 教训
- Navigation BottomNavigationView 切换 tab 时，Fragment 对象保留但 View 被销毁重建
- `onDestroyView` ≠ `onDestroy`，不能在这里释放非 UI 资源（如网络连接、轮询协程）
- 长连接/轮询类 Manager 应该比 View 生命周期更长，在 Fragment 级别持有

原因：Navigation `setupWithNavController` 默认**销毁并重建整个 Fragment 对象**（不仅是 View），Fragment 级别的 lazy manager 也跟着一起死了。

### v2 正确方案：放弃 Navigation tab，改手动 hide/show
- `activity_main.xml`：`NavHostFragment` → `FrameLayout`
- `MainActivity.kt`：手动管理 5 个 tab Fragment 的 hide/show，缓存到 `Map<Int, Fragment>`
- 切 tab 只隐藏/显示，不销毁，远程协助画面和轮询完美保持
- Fragment 恢复逻辑保留（`restoreFragments()` 处理 Activity 重建场景）

### 附带修复：Gradle 内存放大
- `fall-detection-app/gradle.properties`：Xmx 1024m → 1536m
- `~/.gradle/gradle.properties`：全局 Xmx 1536m
- 原因：老人端项目 1024m 编译不动

### 教训
- **Navigation `setupWithNavController` 默认行为是 destroy/recreate Fragment**，不适合需要保持状态的 tab（如远程协助、实时流）
- 需要保持状态的 tab 应用手动 hide/show 或 ViewPager2 + offscreenPageLimit
- `onDestroyView` ≠ `onDestroy`，不能在这里释放非 UI 资源（如网络连接、轮询协程）

## 22:25 K70部署尝试 — 关键教训

### 尝试过程
用户要求直接通过adb在K70上部署diedaobao-server，K70已通过USB连接Mac。

### 遇到的问题

1. **K70没有Termux** — 需要先安装
2. **F-Droid/GitHub全超时** — 无论Mac还是K70都连不上（网络问题）
3. **国内Termux镜像可用** — `mirrors.tuna.tsinghua.edu.cn` 下载APK成功

5. **尝试直接跑Node.js二进制** — 下载了Linux aarch64版但失败
6. **关键教训：Android的动态链接器是 `/system/bin/linker64`，不是Linux的 `/lib/ld-linux-aarch64.so.1`** — 标准Linux二进制无法在Android上运行！
7. **Termux的node包有大量依赖**（libc++, openssl等）— 不能简单adb push单个二进制

### 结论
- **必须通过Termux安装Node.js**，不能直接push二进制
- Termux安装方法：要么adb install成功（需解决签名问题），要么在手机上手动安装
- 也可以考虑在Mac上搭建Termux文件系统然后push进去，但太复杂
- **最简单方案**：用户在K70上手动装Termux（应用商店或文件管理器安装APK），然后adb push项目文件 + adb shell执行
