### 08:00 - 远程协助画面延迟8秒+ 完整根因分析（已读完三端代码）

**用户明确**：不是隧道问题，是画面逻辑问题。滑动没延迟，只有画面帧延迟。

**已读代码（完整帧链路）**：
1. ✅ 老人端 `ScreenCaptureService.kt` — 帧生产+发送
2. ✅ 老人端 `WSClient.kt` — `pushAssistFrameBinary()` 实现
3. ✅ 后端 `ws.js` — `handleBinaryMessage()` 帧转发
4. ✅ 子女端 `WSClient.kt` — `handleBinaryFrame()` 帧接收
5. ✅ 子女端 `RemoteAssistManager.kt` — `handleWSFrame()` 帧渲染
6. ✅ 子女端 `RemoteAssistFragment.kt` — UI显示

**3个延迟瓶颈（确认）**：

**瓶颈1：帧发送在captureHandler单线程上执行**
- `setOnImageAvailableListener(listener, captureHandler)` → processFrame()在单线程HandlerThread
- `processFrame()` → `uploadFrameAsync()` → `WSClient.pushAssistFrameBinary()`
- OkHttp `webSocket.send(ByteString)` 虽然非阻塞入队，但如果网络慢→内部发送缓冲区积压→积压随时间增长→延迟持续增大

**瓶颈2：帧率节流逻辑失效**
- `if (now - lastUploadTime < 333ms) return` — 但lastUploadTime在帧处理完后才更新
- 如果单帧处理+发送耗时>333ms，节流条件永远不触发，每帧都处理
- 实际帧率 = 1/单帧总时间，而不是3fps

**瓶颈3：子女端无帧丢弃**
- `handleWSFrame()` 每帧都解码+post到UI显示，没有"只显示最新帧"逻辑
- 如果WS接收缓冲区有积压帧，会先显示旧帧→延迟感知
- `BitmapFactory.decodeByteArray` + `Handler.post` 每帧都执行

**为什么滑动没延迟**：
- 触控信令是小JSON(~200字节)，同一WS连接全双工发送
- 小消息即使排队也在大帧间隙发出去，几乎无延迟

**延迟计算模型**：
- 帧生产速度 > 网络发送速度 → OkHttp内部发送缓冲区积压
- 积压随时间线性增长 → 延迟从1秒逐渐涨到8秒+
- 关键：没有"丢弃旧帧只发最新帧"机制

**优化方向（按影响排序）**：
1. **发送端帧丢弃**：发送前检查是否有更新帧，只发最新帧（延迟8s→实时）
2. **生产端独立线程**：pushAssistFrameBinary()不阻塞captureHandler
3. **子女端只渲染最新帧**：丢弃待处理旧帧
4. **动态帧率**：网络差时自动降帧率
5. **降低帧大小**：240p / JPEG质量15%
6. **帧号+时间戳诊断日志**：精准定位延迟环节

**验证方法（不改代码）**：
```bash
adb shell "logcat -s ScreenCaptureService | grep '帧'"
```
如果帧间隔从333ms逐渐变成2-5秒 → 确认发送阻塞+积压

**用户要求**：只分析不改代码，分析结果已展示给用户。

### 08:00 - 远程协助画面延迟8秒+ 完整根因分析（已读完三端代码）

**用户明确**：不是隧道问题，是画面逻辑问题。滑动没延迟，只有画面帧延迟。

**已读代码（完整帧链路）**：
1. ✅ 老人端 `ScreenCaptureService.kt` — 帧生产+发送
2. ✅ 老人端 `WSClient.kt` — `pushAssistFrameBinary()` 实现
3. ✅ 后端 `ws.js` — `handleBinaryMessage()` 帧转发
4. ✅ 子女端 `WSClient.kt` — `handleBinaryFrame()` 帧接收
5. ✅ 子女端 `RemoteAssistManager.kt` — `handleWSFrame()` 帧渲染
6. ✅ 子女端 `RemoteAssistFragment.kt` — UI显示

**3个延迟瓶颈（确认）**：

**瓶颈1：帧发送在captureHandler单线程上执行**
- `setOnImageAvailableListener(listener, captureHandler)` → processFrame()在单线程HandlerThread
- `processFrame()` → `uploadFrameAsync()` → `WSClient.pushAssistFrameBinary()`
- OkHttp `webSocket.send(ByteString)` 虽然非阻塞入队，但如果网络慢→内部发送缓冲区积压→积压随时间增长→延迟持续增大

**瓶颈2：帧率节流逻辑失效**
- `if (now - lastUploadTime < 333ms) return` — 但lastUploadTime在帧处理完后才更新
- 如果单帧处理+发送耗时>333ms，节流条件永远不触发，每帧都处理
- 实际帧率 = 1/单帧总时间，而不是3fps

**瓶颈3：子女端无帧丢弃**
- `handleWSFrame()` 每帧都解码+post到UI显示，没有"只显示最新帧"逻辑
- 如果WS接收缓冲区有积压帧，会先显示旧帧→延迟感知
- `BitmapFactory.decodeByteArray` + `Handler.post` 每帧都执行

**为什么滑动没延迟**：
- 触控信令是小JSON(~200字节)，同一WS连接全双工发送
- 小消息即使排队也在大帧间隙发出去，几乎无延迟

**延迟计算模型**：
- 帧生产速度 > 网络发送速度 → OkHttp内部发送缓冲区积压
- 积压随时间线性增长 → 延迟从1秒逐渐涨到8秒+
- 关键：没有"丢弃旧帧只发最新帧"机制

**优化方向（按影响排序）**：
1. **发送端帧丢弃**：发送前检查是否有更新帧，只发最新帧（延迟8s→实时）
2. **生产端独立线程**：pushAssistFrameBinary()不阻塞captureHandler
3. **子女端只渲染最新帧**：丢弃待处理旧帧
4. **动态帧率**：网络差时自动降帧率
5. **降低帧大小**：240p / JPEG质量15%
6. **帧号+时间戳诊断日志**：精准定位延迟环节

adb shell "logcat -s ScreenCaptureService | grep '帧'"
```
如果帧间隔从333ms逐渐变成2-5秒 → 确认发送阻塞+积压
