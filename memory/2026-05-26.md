### 16:27 - "查看位置"无响应根因分析与修复方案

**问题现象**：
- 子女端点击"查看位置"后无反应，老人端未收到位置请求

**根因定位**（基于 server.js 代码分析）：

1. **`/request-elder-location` 端点缺少 HTTP 降级逻辑**
   - 代码位置：server.js 第610-627行
   - 问题：调用 `sendToUser(elderId, ...)` 失败时（wsPushed=false），只写DB标记，没有触发HTTP轮询降级
   - 对比：ws.js 中的 location_request 处理器有降级逻辑（写DB标记让老人端HTTP轮询拉取）

2. **老人端未实现 HTTP 轮询拉取请求**
   - 预期：老人端应定期轮询 `/location-sync?action=poll_pull`
   - 实际：可能未实现或轮询间隔过长

3. **子女端未处理 `wsPushed=false` 的情况**
   - 预期：如果WS推送失败，应降级到HTTP轮询查询位置
   - 实际：可能只依赖WS推送，失败后就卡住

**修复方案**（无需改后端代码）：

1. **老人端添加 HTTP 轮询**（推荐优先做）
   - 在 FallDetectionService 或 WSClient 中添加定时任务
   - 每30秒调用一次 `checkPullRequest()`
   - `checkPullRequest()` 调用 `/location-sync?action=poll_pull`
   - 如果有 `pullLocationRequest` 且状态为 `pending`，则上传位置

2. **子女端添加降级逻辑**（可选）
   - 如果 `wsPushed=false`，启动轮询任务
   - 每5秒查询一次 `/get-status`
   - 直到获取到位置或超时（10次）

**关键代码逻辑**：
```javascript
// server.js 第610-627行（当前代码）
app.post('/request-elder-location', (req, res) => {
  const requestTime = Date.now()
  db.prepare('UPDATE users SET pullLocationRequest=?, pullLocationStatus=? WHERE id=?')
    .run(requestTime, 'pending', elderId)
  
  const wsPushed = sendToUser(elderId, { type: 'location_request', data: { requestTime } })
  
  // ❌ 缺少：如果wsPushed=false，应该确保老人端能通过HTTP轮询拉取到请求
  // ✅ 已有：DB标记已写入，老人端只需实现轮询即可降级
  
  return res.json({ code: 200, success: true, wsPushed })
})
```

**结论**：
- 后端代码逻辑正确，DB标记已写入
- **只需要老人端实现HTTP轮询**即可完成降级
- 不需要修改后端代码

**待执行**：
- [ ] 写老人端 HTTP 轮询代码（checkPullRequest）
- [ ] 编译老人端APK测试
- [ ] 如需要，写子女端降级逻辑
