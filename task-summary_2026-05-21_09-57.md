### 2. 添加 POST /restart 端点代码
在 `server.listen()` 之前插入以下代码：

```javascript
// ========== 服务端自己重启 ==========
app.post('/restart', (req, res) => {
  try {
    const { spawn } = require('child_process');
    const fs = require('fs');
    const path = require('path');
    
    res.json({ code: 200, message: '服务端正在重启...' });
    
    const logFile = path.join(__dirname, 'server.log');
    const out = fs.openSync(logFile, 'a');
    const err = fs.openSync(logFile, 'a');
    
    const child = spawn('nohup', ['node', 'server.js'], {
      cwd: __dirname,
      detached: true,
      stdio: ['ignore', out, err]
    });
    child.unref();
    
    console.log(`✅ 新进程已启动，PID: ${child.pid}`);
    
    setTimeout(() => {
      console.log(`🛑 当前进程退出，新进程已接管`);
      process.exit(0);
    }, 2000);
  } catch (e) {
    res.status(500).json({ code: 500, message: e.message });
  }
});
```

**关键设计**：
- 使用 `nohup` + `detached: true` 启动新进程，避免父进程退出导致子进程被 kill
- 先返回 200 响应，再延迟 2 秒退出当前进程，确保客户端收到响应
- 日志重定向到 `server.log`，避免丢失输出

## 关键推理

1. **为什么用 `nohup` + `detached: true`？**
   - 避免父进程退出时 kill 子进程
   - `detached: true` 使子进程成为新的进程组 leader
   - `child.unref()` 允许父进程独立退出

2. **为什么先返回响应再退出？**
   - 如果先退出，客户端会收到 ECONNRESET
   - 延迟 2 秒足够客户端收到完整响应

3. **为什么不用 `process.exit()` 直接退出？**
   - 需要给新进程足够时间启动
   - 先启动新进程 → 再退出旧进程，实现无缝重启
