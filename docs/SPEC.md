## 一、项目背景与目标

### 1.1 为什么做这个 App

| 数据 | 说明 |
|------|------|
| 中国65岁以上老人跌倒发生率 | 约 30%（每3个老人中有1人每年跌倒） |
| 跌倒是中国老人意外伤害死亡的首位原因 | 每年约 40 万老人因跌倒死亡 |
| 跌倒后1小时内得到救助 | 死亡率降低 50% |
| 空巢/独居老人数量 | 超过 1 亿，且持续增长 |
| 现有方案痛点 | Apple Watch 贵（3000+元）；专业设备操作复杂；无中文老人友好界面 |

**核心价值**：用智能手机（百元机即可）实现专业级跌倒检测，成本低、零门槛、老人自主使用。

### 1.2 项目目标

**终极目标**：一款让老人安心、让子女放心的跌倒检测与紧急救助 App。

**分阶段目标**：

| 阶段 | 目标 | 验证标准 |
|------|------|----------|
| P0 | 跌倒检测准确率 > 85%，误报率 < 20% | SisFall 数据集验证 |
| P1 | 检测后 30 秒内通知紧急联系人 | 实机测试 < 30 秒 |
| P2 | 覆盖主流 Android 设备 | 酷安 + 应用宝上架 |
| P3 | 日活用户 > 1000，7日留存 > 40% | 运营数据 |

### 1.3 参考竞品分析

| 产品 | 检测方式 | 价格 | 优点 | 缺点 |
|------|----------|------|------|------|
| Apple Watch 跌倒检测 | 加速度+陀螺仪+气压计+机器学习 | 3000-8000元 | 准确率高，生态完整 | 贵；仅iOS |
| 三星 Galaxy Watch | 加速度+陀螺仪 | 1500-3000元 | 续航强 | 价格较高；操作复杂 |
| 小米/华米手表 | 加速度+算法 | 300-800元 | 性价比高 | 算法精度一般；无独立App |
| **我们的方案** | 纯手机传感器（加速度+陀螺仪） | 免费/低成本 | 零硬件门槛；老人易用 | 手机必须随身携带 |

**差异化定位**：不做硬件，用已有的手机。老人不一定戴手表，但几乎都带手机。

### 3.2 功能详述

#### F1：跌倒检测（核心引擎）

**技术方案**：阈值法（Phase 1）+ 机器学习（Phase 2）

**Phase 1 阈值法算法**：
```
采样频率：50Hz（每秒50次）
窗口大小：1秒（50个样本）

加速度模值：
  acc_mag = sqrt(ax² + ay² + az²)  [单位：g]

跌倒判定条件（同时满足）：
  条件1：acc_mag 峰值 > 2.5g 且 < 8g（排除假冲击）
  条件2：持续时间 < 1.5秒（跌倒是瞬时冲击）
  条件3：跌倒后 5-30 秒内无明显活动（人在地上不动）
  条件4：陀螺仪姿态角变化 > 60°（从站立到倒地）

误报过滤：
  - 连续2次检测间隔 < 60秒 → 第二次过滤（防止重复报警）
  - 检测到行走/跑步活动 → 降低灵敏度
```

**Phase 2 机器学习方案**：
- 数据集：SisFall（已验证，Apache-2.0 许可）
- 特征：均值、方差、峰值、零交叉率、频域特征（FFT）
- 模型：随机森林（轻量，嵌入式友好）→ TensorFlow Lite
- 部署：Android TFLite（本地推理，保护隐私）

#### F2：紧急联系人管理

- 支持录入 1-3 名紧急联系人
- 每人包含：姓名、电话、与老人关系、接收通知方式
- 通知方式：电话直拨 > 企业微信 Webhook
- 子女端无需安装 App，接电话/看微信即可

#### F3：紧急通知（三通道，100% 免费）

**触发时机**：跌倒确认后（30秒倒计时无人响应）

```
第一通道：系统电话直拨（100% 免费，不依赖网络）
├── Intent.ACTION_CALL → 直接拨打电话，走用户手机卡
├── 无需第三方服务
├── 若未接，30秒后拨第二联系人，最多重试3轮
└── 覆盖：手机有信号就行

├── SmsManager.sendTextMessage() → 直接发送短信
├── 不经过任何第三方，走用户自己的手机号发短信
├── 包含：GPS 位置链接 + 跌倒时间
├── 短信费用：包含在用户手机套餐内（国内短信基本都免费/无限量）
```

## 四、技术方案

### 4.1 技术选型：Kotlin 原生 Android

**为什么选 Kotlin 原生而不是 React Native/Flutter**：

| 对比项 | Kotlin 原生 | React Native | Flutter |
|--------|------------|--------------|---------|
| 传感器 50Hz 采样 | ✅ 直接调用 SensorManager | ⚠️ 需写 Native Module 桥接 | ⚠️ 需写 Platform Channel |
| Android 后台保活 | ✅ Foreground Service 原生支持 | ❌ 后台容易被杀 | ❌ 后台容易被杀 |
| 性能 | ✅ 最优 | ⚠️ JS Bridge 有延迟 | ✅ 好 |
| TFLite 集成 | ✅ 官方 SDK 原生支持 | ⚠️ 需 Native Module | ✅ 有插件 |
| 开发速度 | ✅ 单平台最快 | ⚠️ 跨平台才更快 | ⚠️ 跨平台才更快 |
| APK 体积 | ✅ 最小（5-10MB） | ❌ 包含 JS 引擎（15-25MB） | ❌ 包含 Flutter 引擎（15-20MB） |
| 上架复杂度 | ✅ 最低 | ⚠️ 需处理原生桥接 | ⚠️ 需处理原生桥接 |

**结论**：只做 Android → Kotlin 原生是唯一最优解。传感器高频采样 + 后台保活是跌倒检测的硬性需求，RN/Flutter 在这两个点上都有明显短板。

### 4.2 传感器集成方案（Android SensorManager）

```kotlin
// Kotlin 原生传感器采集（50Hz）
class FallSensorManager(context: Context) {
    private val sensorManager = context.getSystemService(Context.SENSOR_SERVICE) as SensorManager
    private val accelerometer = sensorManager.getDefaultSensor(Sensor.TYPE_ACCELEROMETER)
    private val gyroscope = sensorManager.getDefaultSensor(Sensor.TYPE_GYROSCOPE)
    
    // 采样率：SENSOR_DELAY_GAME ≈ 50Hz，足够跌倒检测
    // 不用 SENSOR_DELAY_FASTEST（200Hz）→ 省电
    private val samplingRateUs = 20_000  // 50Hz = 20ms 间隔
    
    fun start() {
        sensorManager.registerListener(
            sensorEventListener, accelerometer, samplingRateUs
        )
        sensorManager.registerListener(
            sensorEventListener, gyroscope, samplingRateUs
        )
    }
    
    private val sensorEventListener = object : SensorEventListener {
        override fun onSensorChanged(event: SensorEvent) {
            val ax = event.values[0]  // m/s² → 转换为 g（除以 9.81）
            val ay = event.values[1]
            val az = event.values[2]
            val accMag = sqrt(ax*ax + ay*ay + az*az) / 9.81f
            // 传入检测引擎
            fallDetector.feed(accMag, timestamp)
        }
    }
}
```

### 4.3 算法模块架构

```
┌─────────────────────────────────────────────────────┐
│                   传感器数据流                        │
│                                                     │
│  [加速度计 50Hz] ──┐                                │
│                    ├──→ [滑动窗口 buffer] ──→ [特征提取]
│  [陀螺仪 50Hz]  ───┘                                │
│                         ↓                           │
│                   [阈值法预判]  (Phase 1)            │
│                         │                           │
│                         ↓                           │
│                   [ML模型推理]  (Phase 2)            │
│                         │                           │
│                         ↓                           │
│              [跌倒事件] / [正常活动] / [误报]        │
└─────────────────────────────────────────────────────┘
```

**数据流向**：
1. 原生层每秒采样 50 次，加速度+陀螺仪
2. 每 1 秒（50 样本）计算一次滑动窗口特征
3. Phase 1：阈值法预判，acc > 2.5g + 姿态异常 → 触发确认
4. Phase 2：随机森林/TFLite 推理，输出跌倒概率
5. 跌倒事件 → 触发 30 秒确认倒计时
6. 30 秒无取消 → 触发紧急联系人通知

### 4.4 后端架构：无后端（纯离线优先）

**设计原则**：跌倒检测 + 紧急通知完全在手机本地完成，不依赖任何服务器。

```
┌─────────────────────────────────────────────┐
│            红米 K90（全部在手机上）            │
├─────────────────────────────────────────────┤
│  SensorManager（50Hz 传感器采集）             │
│       ↓                                     │
│  FallDetector.kt（阈值法/ML 本地推理）        │
│       ↓                                     │
│  确认弹窗（30秒倒计时）                       │
│       ↓                                     │
│  Intent.ACTION_CALL（系统电话直拨）           │
│       ↓                                     │
│  SmsManager（系统短信，走手机卡）             │
│       ↓                                     │
│  企业微信 Webhook（GPS位置推送）              │
├─────────────────────────────────────────────┤
│  Room SQLite（联系人、事件记录、本地存储）      │
└─────────────────────────────────────────────┘
```

**为什么不需要后端**：
- 传感器数据采集 → 本地 SensorManager
- 跌倒判定 → 本地算法/TFLite
- 紧急通知 → 系统电话 API + 企业微信 Webhook（不经过自己的服务器）
- 数据存储 → 本地 Room SQLite
- GPS → 系统 LocationManager

**零服务器成本的代价**：无法做远程监控面板（子女看不到老人实时状态）。Phase 3 如果需要，再引入免费后端方案（Supabase Free Tier / Firebase Free Tier）。

### 9.2 模型训练方案

#### 两阶段训练策略

**第一阶段：基于公开数据的冷启动模型**

```
数据集：SisFall（4382段）
    ↓
特征工程（窗口法）
    ↓
训练：随机森林（Random Forest）
    ↓
验证：留出法（80%训练 / 20%测试）
    ↓
指标：准确率、召回率、F1-score
    ↓
导出：TFLite 模型（< 1MB）
```

**第二阶段：基于真实用户数据的个性化模型**

```
Phase 3 积累真实跌倒事件（用户同意授权）
    ↓
定期用真实数据重训练（月度更新）
    ↓
可选：Per-user 个性化阈值（根据个人活动模式微调）
```

#### 特征工程（关键）

每个 1 秒滑动窗口（50 样本 @ 50Hz）计算以下特征：

```python
# 频域特征（FFT，检测周期性活动）
features = {
    # 时域（Time Domain）
    "acc_mean": np.mean(acc_mag),           # 加速度均值
    "acc_std": np.std(acc_mag),             # 标准差（检测冲击）
    "acc_max": np.max(acc_mag),             # 峰值（核心跌倒特征）
    "acc_min": np.min(acc_mag),             # 谷值
    "acc_range": np.max(acc_mag) - np.min(acc_mag),  # 动态范围
    "acc_peak_count": count_peaks(acc_mag),  # 峰值数量
    
    # 陀螺仪（姿态变化）
    "gyro_mean": np.mean(gyro_mag),         # 角速度均值
    "gyro_max": np.max(gyro_mag),            # 角速度峰值
    
    # 组合特征
    "impact_ratio": acc_max / (acc_mean + eps),  # 冲击比
    "post_impact_quiet": is_quiet_after(窗口后半段),  # 跌倒后是否静止
    
    # 频域（FFT）
    "fft_energy": np.sum(np.abs(fft(acc_mag))**2),  # FFT能量
    "fft_peak_freq": np.argmax(np.abs(fft(acc_mag))),  # 主频率
}
```

#### 模型选型

| 模型 | 准确率 | 参数量 | 推理速度 | 推荐 |
|------|--------|--------|----------|------|
| 随机森林（100树） | ~88% | ~1MB | < 5ms | **Phase 1 首选** |
| LSTM（单层） | ~91% | ~500KB | < 10ms | Phase 2 |
| 1D-CNN | ~93% | ~300KB | < 5ms | Phase 2 |
| TinyML 版 ResNet | ~95% | < 100KB | < 3ms | 长期 |

**选定**：随机森林（Phase 1）→ 1D-CNN（Phase 2），均为 TFLite 格式，Android 原生集成。

#### 验证指标

| 指标 | 定义 | 目标 |
|------|------|------|
| **召回率（Recall）** | 真实跌倒中被检测出的比例 | **≥ 90%**（漏报比误报更危险） |
| **精确率（Precision）** | 检测为跌倒中真实跌倒的比例 | ≥ 70%（允许一定误报，有两步确认兜底） |
| **F1-score** | 召回率和精确率的调和平均 | ≥ 80% |
| **检测延迟** | 跌倒发生到检测判定的时间 | < 2 秒 |

> ⚠️ **重要**：跌倒检测的召回率（不漏报）比精确率（不误报）更重要。一次漏报的代价可能是生命，几次误报只是打扰。

### 9.4 冷启动问题解决方案

**问题**：Phase 1 没有真实跌倒数据，ML 模型效果存疑。

**解决方案**：

| 策略 | 说明 | 实施时间 |
|------|------|----------|
| **Phase 1 纯阈值法** | 不上 ML，用阈值法足够跑通核心链路 | Week 1-4 |
| **SisFall 数据验证** | 用公开数据集证明算法原理可行 | Week 2 |
| **主动录制真实数据** | 组织团队+志愿者录制，覆盖多种跌倒场景 | Week 3-4 |
| **公开数据集 + 自录制合并训练** | Phase 2 的 ML 模型用合并数据训练 | Phase 2 |
| **用户反馈驱动的阈值调整** | 误报事件 → 自动放宽阈值 | Phase 2 |
| **个性化模型** | 每个用户积累自己的活动数据 → 个性化阈值 | Phase 3 |

**优先级排序**：

### 9.5 手机放置位置与姿态标准化

**核心问题**：老人把手机放在口袋（正面/侧面）、背包、手提包、腰带——不同位置传感器 XYZ 轴朝向完全不同。

**解决方案**：

```python
# 1. 计算加速度模值（去除方向依赖）
acc_mag = sqrt(ax² + ay² + az²)  # 与方向无关

# 2. 如果用姿态信息（陀螺仪），需要重力参考
# 重力向量始终指向地面，可用于判断手机朝向
gravity = acc_mag[最近1秒均值]  # 低通滤波去除冲击

# 3. 跌倒特征用模值而非原始轴数据
# 跌倒是全身运动，模值能反映冲击力大小，与放置位置无关
```

**手机放置测试矩阵**：
| 放置位置 | 行走测试 | 快速坐下 | 模拟跌倒 | 是否可行 |
|----------|----------|----------|----------|----------|
| 正面口袋 | ✅ | ✅ | ✅ | 推荐 |
| 侧面口袋 | ✅ | ✅ | ✅ | 推荐 |
| 背包（胸前） | ✅ | ✅ | ⚠️ | 可行 |
| 背包（背部） | ✅ | ⚠️ | ⚠️ | 可行但需单独标定 |
| 手提包 | ✅ | ✅ | ⚠️ | 可行 |
| 腰带夹 | ✅ | ✅ | ✅ | 推荐 |

## 十三、耗电优化方案（关键）

老人手机电量焦虑是真实的。后台 50Hz 传感器采样会很耗电。

**优化策略**：

| 策略 | 说明 | 预期省电 |
|------|------|----------|
| **事件驱动采样** | 静止时降为 5Hz，检测到运动后升至 50Hz | 节省 80% |
| **充电时全速** | 充电中不受限，正常 50Hz | — |
| **勿扰时段** | 睡眠时降为 10Hz（老人睡觉不会跌倒…也可能） | 节省 60% |
| **自适应采样** | 根据过去 1 小时活动量调整频率 | 智能调节 |
| **批量处理** | 不每帧计算，攒 50 帧（1秒）统一处理 | 减少 CPU 唤醒 |
| **TFLite 量化** | int8 量化模型，减少推理计算量 | 推理快 3x |

**耗电目标**：
- 后台运行 < 3%/小时（普通手机 4000mAh → 可用 33 小时）
- 实际测试标准：红米 K90 装满电，后台运行 24 小时，剩余 > 72%
