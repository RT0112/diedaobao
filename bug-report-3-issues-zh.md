## Bug 2: 查看位置功能卡住（一直显示"正在获取"）

**问题描述**：
子女端点击"查看位置"按钮后，如果等待一段时间（比如几分钟）再点击，界面会一直显示"正在获取"，无法获取到位置，也不会超时失败。

**复现步骤**：
1. 打开子女端App
2. 等待几分钟（不操作）
3. 点击"查看位置"按钮
4. 观察：一直显示"正在获取"，永远不会超时或返回结果

**期望行为**：
- 应该有超时机制（比如30秒后提示"获取位置超时"）
- 或者能够正常获取到位置

**实际行为**：
一直显示"正在获取"，卡住不动。

**可能原因**：
1. 后端WebSocket连接已断开，但UI状态没有重置
2. 位置请求没有设置超时
3. 后端server没有正确处理延迟的位置请求
4. 状态机没有处理"等待超时"的情况

**需要排查的文件**：
- 子女端：位置请求相关的ViewModel/Repository
- 后端server：位置请求的超时处理
- WebSocket连接状态管理

## Bug 3: 强制弹窗设置不生效

**问题描述**：
在老人端App的设置里，打开了"强制弹窗"的开关，但实际使用时并没有弹出窗口（或者说弹窗没有正常显示）。

**复现步骤**：
1. 打开老人端App
2. 进入设置
3. 打开"强制弹窗"开关
4. 触发远程协助（比如子女端请求查看位置或拍照）
5. 观察：老人端没有弹出窗口（或者弹窗没有正确显示）

**期望行为**：
打开强制弹窗开关后，子女端发起远程协助请求时，老人端应该弹窗提示用户（即使App在后台）。

**实际行为**：
强制弹窗开关打开了，但没有生效（老人端没有弹窗）。

**可能原因**：
1. 设置没有正确保存到SharedPreferences
2. 读取设置时用了错误的key
3. 弹窗逻辑没有检查这个设置
4. 高版本Android的悬浮窗权限问题（需要SYSTEM_ALERT_WINDOW权限）
5. 弹窗代码没有正确处理"强制弹窗"的逻辑

**需要排查的文件**：
- 老人端：设置页面（SharedPreferences保存）
- 老人端：弹窗逻辑（读取设置并决定是否弹窗）
- 老人端：悬浮窗权限检查
- 子女端：发起远程协助的请求逻辑
